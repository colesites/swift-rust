# SR0045 — Export path mismatch

A path in `exportPathMap` does not match the dynamic route it points at. The page would be exported to a URL missing one or more parameters, so Swift-Rust refuses to ship the export.

## Why this error occurred

`exportPathMap` is the legacy way to enumerate every URL a static export should produce. Each entry maps a URL path to a page file. For a dynamic route like `app/blog/[post]/comment/[id]/page.tsx`, the exported URL has to fill in both `[post]` and `[id]`, e.g. `/blog/hello/comment/1`. If the entry leaves a segment out — `/blog/hello` instead of `/blog/hello/comment/1` — the export would write a file that does not match the route's URL pattern, and the deployed site would 404 on it.

The framework validates every entry against the route graph at build time and rejects mismatches so the failure is caught in CI rather than at the CDN.

## Reproducing the error

```json
{
  "name": "site",
  "output": "export",
  "exportPathMap": {
    "/": { "page": "/" },
    "/blog/nextjs": { "page": "/blog/[post]/comment/[id]" }
  }
}
```

## Error output

```
[ERROR] [SR0045] export path mismatch: /blog/nextjs does not match /blog/[post]/comment/[id]
   --> swift-rust.config.json:4:5
   |
 4 |     "/blog/nextjs": { "page": "/blog/[post]/comment/[id]" }
   |     ^^^^^^^^^^^^^^^ expected /blog/nextjs/comment/<id>; missing [id] segment
   |
  help: include every dynamic segment in the path, e.g. "/blog/nextjs/comment/1".
```

## How to fix it

1. Update the path so it includes a concrete value for every dynamic segment in the target route.
2. Verify the `page` field points at a real file under `app/`.
3. Re-run the build and confirm every entry resolves.

```json
{
  "name": "site",
  "output": "export",
  "exportPathMap": {
    "/": { "page": "/" },
    "/blog/nextjs/comment/1": { "page": "/blog/[post]/comment/[id]" }
  }
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build --static
```

The trace log lists every `exportPathMap` entry it considered and whether each one matched a real route. The list makes it easy to fix the whole map in one pass.

## Common pitfalls

- Catch-all segments (`[...slug]`) must also be filled in. The export URL needs at least one concrete segment.
- Optional catch-alls (`[[...slug]]`) can be matched with the bare path, but the `page` field must still be correct.
- The `page` field is relative to the `pages/` directory in the legacy API, or to the `app/` directory in the new one. Make sure the format matches your router.
- A typo in a `page` value (e.g. `/blog/[post]/comments/[id]` with an extra `s`) produces a different error: the file is not found at all. This error fires only when the path exists but the segments do not line up.

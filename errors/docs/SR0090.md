# SR0090 — Missing environment variable value

A value read from `process.env` was `undefined` at build or request time. The build fails because the missing value is required to compile or to render.

## Why this error occurred

Swift-Rust evaluates `process.env.X` in two distinct phases:

- **Build time.** Server components run during the build to produce static HTML. Any `process.env.X` read at the top level of a server component is evaluated at build time. If `X` is not defined, the build fails.
- **Request time.** Reads inside dynamic functions (route handlers, server actions) happen per-request. If `X` is not defined, the request returns 500.

Client components only see variables prefixed with `SR_PUBLIC_`. A typo (e.g. `NEXT_PUBLIC_FOO` from a Next.js migration) makes the variable invisible on the client and `undefined` in the build.

The framework cannot guess what value you meant, so it stops the build and tells you exactly which variable was missing.

## Reproducing the error

```tsx filename="app/page.tsx"
export default function Page() {
  return <p>API host: {process.env.API_HOST}</p>
}
```

`API_HOST` is not defined in `.env` and is not prefixed for client use.

## Error output

```
error: [SR0090] missing env value: API_HOST is undefined
  --> app/page.tsx:1:33
   |
 1 | export default function Page() {
 2 |   return <p>API host: {process.env.API_HOST}</p>
   |                              ^^^^^^^^^
```

## How to fix it

1. Add the variable to `.env` at the project root.
2. For values that must be inlined into client bundles, prefix the variable with `SR_PUBLIC_` (e.g. `SR_PUBLIC_ANALYTICS_ID`).
3. Verify the env file is loaded: the `bun swift-rust` CLI loads `.env` and `.env.local` automatically.

```bash filename=".env"
API_HOST=https://api.example.com
SR_PUBLIC_ANALYTICS_ID=UA-12345
```

```tsx filename="app/page.tsx"
export default function Page() {
  return <p>API host: {process.env.API_HOST}</p>
}
```

## Diagnosis

```bash
bun swift-rust info --env
```

`--env` prints the merged environment that the build will see, with `SR_PUBLIC_` values flagged.

## Common pitfalls

- `.env.local` is intended for secrets and overrides `.env`. Make sure CI does not commit it.
- Server-only variables that are not prefixed with `SR_PUBLIC_` will be replaced with `undefined` if they leak into a client component. The build warns, but does not fail, on accidental client reads.
- Variables are read at the time of the access, not at startup. If you mutate `process.env` in a module top-level statement, that mutation is visible only inside that module.

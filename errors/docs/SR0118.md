# SR0118 ‚Äî Non-standard NODE_ENV

`NODE_ENV` is treated as a strict enum by Swift-Rust and the wider JavaScript ecosystem: `development`, `production`, or `test`. Anything else is almost always a misconfiguration that breaks dead-code elimination, breaks the `import.meta.env.DEV` constant, and produces confusing behavior in third-party libraries.

## Why this error occurred

The build pipeline reads `NODE_ENV` to decide which code path to bundle, which polyfills to include, and which logging to enable. The value is also exposed to user code through `process.env.NODE_ENV`. When a non-standard value slips in, libraries that switch on `NODE_ENV === 'production'` silently take the "neither" branch and the bundle is shipped with development checks still inside.

The error is a warning rather than a hard error because some teams have legacy tooling that legitimately needs a custom value. Swift-Rust surfaces the issue and lets you decide.

## Reproducing the error

```bash
# .env
NODE_ENV=staging
```

```bash
# .env.local
NODE_ENV=qa
```

```bash
# ~/.zshrc
export NODE_ENV=staging
```

## Error output

```
[WARN] [SR0118] non-standard NODE_ENV: "staging"
  --> .env:1:1
    |
  1 | NODE_ENV=staging
    | ^^^^^^^^^^^^^^^^
    | Swift-Rust only treats "development", "production", and "test" as valid NODE_ENV values.
    |
    help: remove this entry and use a separate variable (e.g. APP_ENV) for custom environments.
```

## How to fix it

1. Remove the `NODE_ENV` line from your `.env*` files. Swift-Rust sets it automatically based on the command you run.
2. If you need to mark a custom environment, introduce a new variable:

```bash
# .env
APP_ENV=staging
```

3. In your app, read `APP_ENV` instead:

```tsx
const env = process.env.APP_ENV ?? "production";
if (env === "staging") {
  console.log("running in staging");
}
```

4. In CI, make sure no shell script exports a non-standard value before invoking `bun swift-rust build`.

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
bun swift-rust doctor --env
```

`--env` prints the resolved environment, including the value of `NODE_ENV` and which file set it.

## Common pitfalls

- `bun` does not silently set `NODE_ENV` for arbitrary scripts. Running `bun script.ts` outside of the Swift-Rust CLI does not make `NODE_ENV=development`.
- `cross-env NODE_ENV=production bun swift-rust build` works, but `cross-env` itself reads from `process.env`, so the same warning fires if the parent shell has a non-standard value.
- Some package managers (`pnpm`) do not normalize `NODE_ENV` across workspaces. Set the value in the workspace root and rely on inheritance.
- Cloud build systems often inject a `NODE_ENV=production` for build steps and a different value for deploy steps. Verify both.

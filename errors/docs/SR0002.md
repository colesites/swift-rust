# SR0002 — Routing error

The router walks `app/` at build time and produces a `matchit` tree of routes. If two files compete for the same route, if a segment shape is invalid, or if a route file is missing the required default export, the router reports this error before the server starts.

## Why this error occurred

The router is the contract between your filesystem layout and the HTTP surface. The framework compiles the `app/` tree into a static routing table, and any ambiguity at that step is unrecoverable at request time, so it has to be caught at startup.

Common root causes:

- A `page.tsx` is empty or exports only a named component, not a default.
- A `route.ts` (API handler) and a `page.tsx` (UI) define the same path.
- A folder like `[id]` sits next to a static segment like `featured`, and both would match `/blog/featured` ambiguously.
- A reserved filename (`_app`, `_document`, `_error`) is used in `app/`; these existed in the legacy Pages Router and are not part of Swift-Rust.

## Reproducing the error

```tsx filename="app/blog/[slug]/page.tsx"
export function Post() {
  return <article>...</article>;
}
```

The file exports a named function, not a default. The router cannot infer the page.

```text filename="app/blog/[slug]/"
page.tsx
route.ts
```

Two files claim the same URL, one for UI and one for an API handler. The router refuses to pick.

## Error output

```
error: [SR0002] routing error: route `GET /blog/[slug]` already exists
  --> app/blog/[slug]/page.tsx:1:1
  --> app/blog/[slug]/route.ts:1:1
   |
   help: remove one of the conflicting files, or split the API into a sub-path
```

```
error: [SR0002] routing error: page file has no default export
  --> app/blog/[slug]/page.tsx:1:1
   |
 1 | export function Post() { ... }
   |        ^^^^ expected `export default function Post()` or `export default Post`
```

## How to fix it

1. Add a `default` export to every page file:

```tsx filename="app/blog/[slug]/page.tsx"
export default function Post({ params }: { params: { slug: string } }) {
  return <article>{params.slug}</article>;
}
```

2. If both a page and an API handler are needed, split them onto different paths. Keep `/blog/[slug]` for the UI and put the API at `/api/blog/[slug]/route.ts`.

3. Resolve static-vs-dynamic collisions by renaming one of the segments or splitting the static case into its own file:

```text filename="app/blog/"
[slug]/page.tsx
featured/page.tsx
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info --tree
RUST_LOG=swift_rust::router=trace bun swift-rust dev
```

`info --tree` prints the compiled route table; pass `--json` to feed it into other tools. The trace log shows the file walk step by step so you can see exactly which file the router is choking on.

## Common pitfalls

- A page file that only re-exports a component is not enough. The router looks for `export default` literally.
- The legacy names `_app.tsx` and `_document.tsx` are silently ignored — they look like files but do not register as routes. Use the root layout in `app/layout.tsx` instead.
- Adding a `loading.tsx` next to a `page.tsx` is fine; both can coexist. The router only complains on hard conflicts.
- `app/` and a legacy `pages/` directory cannot coexist in the same project. The compiler fails fast if both are present.

# SR0005 — Server error

The server catches panics in handlers, route components, and proxy functions and turns them into a 500 response with the error code and message. The same code is logged to stderr with the panic backtrace.

## Why this error occurred

A 500 surfaced to the client is the visible side of an internal panic. The framework is built on Rust + Tokio + Axum, and any panic that escapes an async task is caught at the request boundary and converted into a structured response. The error code is the only stable identifier; the panic message can change between versions.

The three most common triggers are: an `unwrap()` on a `None` value deep in user code, a `reqwest` call that the runtime cannot complete (DNS failure, connection refused), and a `tokio::spawn` inside a handler that does not get awaited before the response is sent.

## Reproducing the error

```tsx filename="app/api/users/route.ts"
export async function GET() {
  const user = await db.user.findFirst();
  return Response.json(user.profile);
}
```

If `user` is `null` (no row matches), the chained `.profile` throws `TypeError: Cannot read properties of null`.

```tsx filename="app/api/echo/route.ts"
export async function POST(request: Request) {
  const body = await request.json();
  return Response.json(body.value.x);
}
```

If `value` is missing from the body, `.x` throws on the server.

## How to fix it

1. Open the file at the line indicated in the backtrace.
2. Replace any `unwrap` / `!` access with an explicit `if (value == null) return new Response(null, { status: 404 })` shape.
3. Validate request bodies with a schema (Zod, Valibot) before reading nested fields.
4. Re-run the dev server to confirm.

```tsx filename="app/api/users/route.ts"
export async function GET() {
  const user = await db.user.findFirst();
  if (!user) {
    return new Response(null, { status: 404 });
  }
  return Response.json(user.profile);
}
```

## Diagnosis

```bash
bun swift-rust dev
RUST_LOG=swift_rust=trace bun swift-rust dev
lsof -i :3000
```

The dev server prints every request with its status and elapsed time. The trace log adds the full panic backtrace. `lsof` confirms whether the port is held by another process.

## Common pitfalls

- A 500 in the browser does not always map to a panic in the handler. Some `fetch` calls return non-2xx responses that get caught by the SWC-generated `error.tsx` boundary. Check the server log to confirm the panic was on the server.
- The server restarts on source changes, but panics in `proxy.ts` are sticky — the proxy is only reloaded on a full server restart, not on file watch.
- A `process.env.X` read at the top of a handler throws if `X` is unset at request time, not at startup. Validate environment at process start (`bun swift-rust doctor --env`) to fail fast.
- Long-running async tasks spawned with `tokio::spawn` that are not awaited may continue past the request lifetime. Their panics surface in the next request's log, not the originating one.

# SR0048 — Failed to find server action

Server actions are React functions that run on the server and are invoked from the client through an opaque action ID. Swift-Rust encrypts the action's source location into that ID using an AES key derived per build. When the client sends a request with an action ID, the server decrypts it, looks up the action, and dispatches the call. If decryption fails or the action cannot be found, the request is rejected with this error.

## Why this error occurred

The action ID encodes the file path, export name, and a bound-argument hash. The encryption key is regenerated on every production build so that an action ID from a previous build cannot be replayed. This is a deliberate security property — it prevents an attacker from calling server actions that have since been deleted or refactored.

The downside is consistency: if you run more than one instance of the Swift-Rust server (load balancer, blue/green deploy, horizontally scaled cluster), every instance must agree on the encryption key. If they do not, a request can be routed to an instance that decrypts the ID to a different file path, or fails to decrypt it at all.

This is exactly the same class of bug as Next.js's `failed-to-find-server-action`, with the variable renamed to `SWIFT_RUST_SERVER_ACTIONS_KEY` for Swift-Rust.

## Reproducing the error

```tsx
"use server";

export async function submitFeedback(form: FormData) {
  await db.feedback.create({ text: form.get("message") });
}
```

```tsx
"use client";
import { submitFeedback } from "./actions";

export function Form() {
  return (
    <form action={submitFeedback}>
      <textarea name="message" />
      <button type="submit">Send</button>
    </form>
  );
}
```

## Error output

```
[ERROR] [SR0048] failed to find server action
  --> POST /submit-feedback
   |
   = the action ID `01HQ3F…` could not be decrypted. this usually means
     the server is using a different encryption key than the one used
     when this client bundle was built.
   = set SWIFT_RUST_SERVER_ACTIONS_KEY on every instance and rebuild.
```

## How to fix it

1. Generate a stable 32-byte AES key and base64-encode it.

   ```bash
   openssl rand -base64 32
   ```

2. Set the same value on every instance and on your build pipeline.

   ```bash
   export SWIFT_RUST_SERVER_ACTIONS_KEY="<base64-key-from-step-1>"
   ```

3. Rebuild the app so the new key is embedded in the client bundle.

   ```bash
   bun swift-rust build
   ```

4. If you deploy to the first-party Swift-Rust platform, enable **skew protection** so requests from a client bundle built against the previous deploy continue to resolve during a rolling update.

   ```json
   {
     "platform": {
       "skewProtection": true
     }
   }
   ```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=server_actions=trace bun swift-rust dev
env | grep SWIFT_RUST_SERVER_ACTIONS_KEY
```

## Common pitfalls

- The key must be set at **build time** as well as runtime — it is embedded into the client bundle so the same key is used to derive the action ID and to decrypt it.
- 16, 24, and 32-byte AES keys are accepted; anything else is rejected. Run `echo $SWIFT_RUST_SERVER_ACTIONS_KEY | base64 -d | wc -c` to confirm the byte length.
- Vercel's `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` is the analogue. If you are migrating a Next.js app, port both the variable name and the build step.
- Skew protection is not a substitute for a stable key — it is a grace period during rollouts. Two instances with different keys will still disagree after the grace window.

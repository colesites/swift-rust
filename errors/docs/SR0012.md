# SR0012 — API routes not allowed in static export

Swift-Rust's static export emits a fully prerendered site with no running server, so route handlers under `app/api/` have nothing to attach to. The build continues, but the offending handlers are skipped and a warning is printed.

## Why this error occurred

A static export works by walking the route graph at build time and writing HTML, RSC payloads, and assets to `dist/`. A `route.ts` is a request-time function — it needs the Swift-Rust server runtime. When the build is configured for `output: 'export'` (or the `--static` flag is passed), the framework cannot ship those endpoints and warns you instead of silently dropping them.

This is a warning rather than an error because the rest of the export is usually valid; you can still deploy the static portion.

## Reproducing the error

```json
{
  "name": "blog",
  "output": "export"
}
```

```tsx
export async function GET() {
  return Response.json({ ok: true });
}
```

## Error output

```
[WARN] [SR0012] api route skipped in static export: app/api/ping/route.ts
   --> app/api/ping/route.ts:1:1
   |
 1 | export async function GET() { ... }
   | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ route handlers require a server build
   |
  help: remove `output: 'export'` from swift-rust.config.json, or move this handler to a non-exported project.
```

## How to fix it

1. Decide whether this project is actually static. If it has a server, drop `output: 'export'` and rebuild.
2. If you only need the API in some environments, move the handlers to a separate Swift-Rust project that is deployed with a server target.
3. If you genuinely want the data available in the static bundle, pre-render it during the build and read the resulting JSON from a server component instead.

```json
{
  "name": "blog"
}
```

```tsx
export async function GET() {
  return Response.json({ ok: true });
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build --static
```

The trace log lists every `route.ts` it encountered and whether each one was included or skipped, so you can confirm none of your endpoints were silently dropped.

## Common pitfalls

- The warning is per file, but the build still succeeds. A green build does not mean your API shipped.
- `revalidate` on a server component is not a substitute for a route handler in a static export.
- Route handlers under `app/api/` that are only used in `notFound()` calls are still flagged.
- This warning is distinct from `SR0044`, which fires when `rewrites`/`redirects`/`headers` are also defined under `output: 'export'`.

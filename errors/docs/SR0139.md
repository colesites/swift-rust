# SR0139 — Rewrite targets auto-export or fallback dynamic route

A rewrite in `swift-rust.config.json` points at a route that is either auto-statically optimized or a fallback SSG page. The route table cannot resolve the dynamic parameters on the client, so the rewrite silently does the wrong thing. The framework prints this warning and skips the rewrite.

## Why this error occurred

Rewrites are applied in two places: the server (when the request first lands) and the client router (when navigating). For a non-dynamic target, both layers agree on the new path. For a dynamic target, the client router cannot reconstruct the dynamic segment values without a server roundtrip, so it falls back to the original URL — which means the rewrite only works for the initial request, not for client-side navigation.

The framework's `output: 'export'` mode and `fallback: true` mode both produce dynamic routes with no server. The rewrite cannot work in that world, so the framework warns and continues.

The warning is `warning` severity, not `error`, because the rewrite is still valid for the initial render — only client-side navigation breaks. The CLI surfaces it so you can fix the target rather than ship a broken navigation experience.

## Reproducing the error

```json
{
  "rewrites": [
    {
      "source": "/blog/:slug",
      "destination": "/posts/:slug"
    }
  ]
}
```

If `app/posts/[slug]/page.tsx` is prerendered (or exported as a static fallback), the rewrite fails to navigate on the client.

## Error output

```
[WARN] [SR0139] rewrite targets auto-export or fallback dynamic route
  --> swift-rust.config.json:4:7
    |
  4 |       "destination": "/posts/:slug"
    |       ^^^^^^^^^^^^^
    | the target `/posts/:slug` is a fallback or auto-exported route. client-side navigation will not apply this rewrite.
    |
    help: prerender the target path, or switch to a non-dynamic destination.
```

## How to fix it

1. Prerender every concrete path the rewrite can hit, so the target is no longer a fallback:

```tsx
// app/posts/[slug]/page.tsx
export async function generateStaticParams() {
  const slugs = await db.posts.listSlugs();
  return slugs.map((slug) => ({ slug }));
}
```

2. Or switch the rewrite to a non-dynamic target:

```json
{
  "rewrites": [
    {
      "source": "/blog",
      "destination": "/posts"
    }
  ]
}
```

3. Or opt the target out of static optimization by reading dynamic data inside the handler:

```tsx
import { headers } from "swift-rust/headers";

export default async function Page({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
  const h = await headers();
  return <Post slug={slug} ts={h.get("x-request-ts")} />;
}
```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=router=trace bun swift-rust dev
```

`router=trace` prints the resolved target for every rewrite and flags ones that hit a fallback or auto-exported route.

## Common pitfalls

- The warning is per-rewrite, not per-config. A config that has 10 rewrites and one bad target will print one warning.
- A rewrite to a static page works fully — only dynamic targets are affected.
- `output: 'export'` in the config marks every route as auto-exported, so rewrites to dynamic paths always fail. This is a fundamental limitation, not a bug.
- The error names the destination, not the source. If you have many rewrites pointing at the same dynamic target, you will see one warning per rewrite.

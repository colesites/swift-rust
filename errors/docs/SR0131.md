# SR0131 — Public _swift-rust folder conflict

The `public/` directory in your project contains a folder named `_swift-rust`, which collides with the framework's internal directory of the same name. Static files served from `public/_swift-rust` would shadow the framework's own bundles, so the build refuses to start.

## Why this error occurred

Swift-Rust writes its built assets to a reserved directory whose name is internal to the framework (the equivalent of `/_next` in Next.js). Anything under that path is handled by the framework and never served from `public/`. If a folder with that reserved name shows up in `public/`, the routing layer cannot tell the two apart.

The check fires at build time and at start time. Once the conflict is resolved, the build proceeds normally.

## Reproducing the error

```
project/
├── public/
│   └── _swift-rust/
│       └── static/
│           └── header.png
├── app/
│   └── page.tsx
└── swift-rust.config.json
```

## Error output

```
[ERROR] [SR0131] public _swift-rust folder conflict
  --> public/_swift-rust
    |
    | the directory `public/_swift-rust` conflicts with the framework's reserved asset path.
    |
    help: rename the folder or remove it. Swift-Rust owns the `_swift-rust` namespace.
```

## How to fix it

1. Rename the conflicting directory:

```bash
mv public/_swift-rust public/assets
```

2. Update any references in your code:

```tsx
<img src="/assets/static/header.png" alt="header" />
```

3. Re-run the build:

```bash
bun swift-rust build
```

## Diagnosis

```bash
bun swift-rust doctor
ls public/
```

`ls public/` shows whether the offending directory is present. The doctor output lists the resolved public directory and the reserved paths that the framework owns.

## Common pitfalls

- The check is case-sensitive. `_swift-rust` and `_Swift-Rust` are both rejected on most filesystems because the underlying case is normalized.
- Dumping a build directory into `public/` is the most common source of the conflict. Use `build/` as the destination, not `public/`.
- A symlink pointing into the framework's internal directory triggers the same check.
- A backup or VCS operation that restored an old `public/_swift-rust` from history will re-trigger the error.

# SR0117 — Node.js module imported in the Swift-Rust edge runtime

The Swift-Rust server runtime can run handlers in two modes: the default Rust-based "edge" runtime (built on axum + V8 isolate) and the Node.js runtime. The edge runtime does not bundle Node.js builtins, and importing one fails the request with a module-resolution error.

## Why this error occurred

Swift-Rust's edge runtime is a sandboxed JavaScript environment. It implements the Web standard platform (fetch, streams, `crypto.subtle`, `URL`, `TextEncoder`, ...) but does not ship Node.js builtins like `fs`, `path`, `crypto`, `child_process`, or `process` (with the exception of a small allowlist such as `process.env` at build time). When the bundler resolves an import of a Node-only module, the build fails with this error.

The error can be triggered transitively. A library that you import may pull in a Node-only module as a side effect; the bundler only knows the top-level import, so the message names the package the user wrote, not the package that actually requested Node.

## Reproducing the error

```tsx
import crypto from "node:crypto";
import { NextResponse } from "swift-rust/server";

export const runtime = "edge";

export async function GET() {
  const hash = crypto.createHash("sha256").update("hello").digest("hex");
  return NextResponse.json({ hash });
}
```

## Error output

```
[ERROR] [SR0117] node.js module imported in edge runtime: node:crypto
  --> app/api/hash/route.ts:1:1
    |
  1 | import crypto from "node:crypto";
    | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    | the edge runtime does not provide `node:crypto`; use the Web Crypto API instead.
    |
    help: switch the runtime to `nodejs`, or replace `node:crypto` with the global `crypto.subtle` digest API.
```

## How to fix it

1. Switch the handler to the Node.js runtime if the dependency genuinely needs Node builtins:

```tsx
import crypto from "node:crypto";
import { NextResponse } from "swift-rust/server";

export const runtime = "nodejs";

export async function GET() {
  const hash = crypto.createHash("sha256").update("hello").digest("hex");
  return NextResponse.json({ hash });
}
```

2. If you want to stay on the edge runtime, replace the import with the Web Crypto API:

```tsx
import { NextResponse } from "swift-rust/server";

export const runtime = "edge";

export async function GET() {
  const buf = new TextEncoder().encode("hello");
  const digest = await crypto.subtle.digest("SHA-256", buf);
  const hash = Array.from(new Uint8Array(digest))
    .map((b) => b.toString(16).padStart(2, "0"))
    .join("");
  return NextResponse.json({ hash });
}
```

3. Audit `package.json` for transitive Node-only deps. `bun pm ls` shows the dep tree.

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

`swift-rust info` lists every handler and the runtime it resolves to, which makes it easy to see where edge vs. nodejs differs from what you intended.

## Common pitfalls

- `node:async_hooks`, `node:worker_threads`, and `node:cluster` are unsupported in the edge runtime even with `runtime = "nodejs"` if the host is configured for the static export target.
- The `crypto` global (lowercase, no `node:` prefix) is provided by both runtimes but exposes different surfaces: `crypto.subtle` in the edge, the full `crypto` namespace in Node.
- Some packages ship dual ESM/CJS builds and pick the Node one when bundled. Pin the import path or use a conditional export map.
- The `process` global is shimmed in the edge runtime with `process.env` populated at build time only. Reading `process.cwd()` at request time throws.

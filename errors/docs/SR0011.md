# SR0011 — API route response exceeded size limit

A route handler in `app/api/` finished writing a body that exceeded the default 4 MB response ceiling. Swift-Rust enforces the cap so a single misbehaving endpoint cannot exhaust the request worker.

## Why this error occurred

The Rust HTTP runtime backing Swift-Rust buffers route handler bodies in memory before flushing them to the client. Hard-coded 4 MB is the safety limit. A handler that returns a `Response.json(...)` with a huge array, or that returns a `Buffer`/`ArrayBuffer` of encoded data, hits the ceiling.

The limit is configurable per route because the default exists for protection, not for performance. Disabling it in a long-lived request handler can starve the worker pool, so only do it when you have measured the cost.

## Reproducing the error

```tsx
export async function GET() {
  const huge = "x".repeat(5 * 1024 * 1024);
  return new Response(huge);
}
```

## Error output

```
[ERROR] [SR0011] api route response exceeded size limit: 5.24 MB > 4 MB cap
  --> app/api/export/route.ts:5:18
   |
  5 |   return new Response(huge);
    |            ^^^^ body of 5,242,881 bytes exceeds the 4 MB route handler ceiling
  |
  help: stream the body in chunks, or move the payload to storage and return a URL.
```

## How to fix it

1. Switch the response to a `ReadableStream` so Swift-Rust does not need to buffer the whole body in memory.
2. If the data is large binary content (images, exports), write it to a Swift-Rust storage adapter and return a signed URL.
3. For internal batch endpoints where you have measured the impact, opt out per route with the `responseLimit` config.

```tsx
export const config = {
  api: { responseLimit: "16mb" },
};

export async function GET() {
  const stream = new ReadableStream({
    start(controller) {
      for (let i = 0; i < 100; i++) {
        controller.enqueue(new TextEncoder().encode(chunk(i)));
      }
      controller.close();
    },
  });
  return new Response(stream, {
    headers: { "content-type": "text/csv" },
  });
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

The `trace` log prints the actual byte count and the handler that produced it, which makes it easy to confirm whether the limit is genuinely being hit or a downstream proxy is rewriting the body.

## Common pitfalls

- The cap is on the encoded body, not the JSON value. Stringifying a 3 MB array can push the wire size over 4 MB once headers and escaping are counted.
- A `Response` with a `Blob` or `FormData` body still goes through the same cap. Streams are the only shape that bypasses buffering.
- Setting `responseLimit: false` disables the cap entirely. Prefer a numeric value such as `"8mb"` to keep an upper bound.
- The cap applies per request, not per connection. Long-lived SSE / WebSocket upgrades are not affected.

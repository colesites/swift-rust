# SR0019 — Built-in CSS support disabled

A custom CSS plugin was added to `swift-rust.config.json`, which opts the project out of Swift-Rust's built-in CSS / Sass / PostCSS pipeline. The build continues, but the deprecation is logged because the legacy plugin shim will be removed in a future major version.

## Why this error occurred

In the early Swift-Rust alpha, projects had to install `@zeit/next-css` and friends to get any CSS support. The compiler now handles global CSS, CSS Modules, Sass, and PostCSS out of the box, so the plugin is redundant. Detecting the plugin in the config triggers this deprecation so the migration path is clear: delete the entry, and everything keeps working.

The deprecation is non-fatal so existing projects can migrate at their own pace.

## Reproducing the error

```json
{
  "name": "site",
  "experimental": {
    "plugins": ["@zeit/next-css"]
  }
}
```

## Error output

```
[WARN] [SR0019] built-in CSS support disabled by a custom plugin
   --> swift-rust.config.json:3:17
   |
 3 |     "plugins": ["@zeit/next-css"]
   |                 ^^^^^^^^^^^^^^^^^^ legacy plugin; remove it to re-enable built-in CSS handling
   |
  help: delete the entry. Swift-Rust's built-in CSS pipeline covers global CSS, modules, and Sass.
```

## How to fix it

1. Open `swift-rust.config.json` and remove the legacy plugin entry.
2. If the plugin was wiring a non-default PostCSS transform, move it under `experimental.postcss`.
3. Re-run the build and confirm the deprecation is gone.

```json
{
  "name": "site"
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The trace log shows which plugin was detected and which file system paths it claimed to handle, so you can confirm whether the plugin is still doing real work or just sitting idle.

## Common pitfalls

- Some third-party starters register the plugin by default. Re-running `create-swift-rust` with the current CLI drops the entry.
- `@zeit/next-sass` is the Sass equivalent; the fix is the same.
- If you genuinely need to eject the CSS pipeline, set `experimental.cssLoaders` instead. The deprecation will not fire in that case.
- The deprecation is paired with `SR0016` when a Babel config is also present.

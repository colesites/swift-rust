# SR0033 — Custom error without a 404 page

A custom `app/error.tsx` was added to the project but no `app/not-found.tsx` was provided. Without a 404 page, the framework falls back to a generic not-found UI that is not styled to match the rest of the app.

## Why this error occurred

In the legacy pages router, `pages/_error.tsx` handled every error status, including 404. In the app router, errors and not-found responses are different primitives: `error.tsx` catches thrown errors during rendering, while `not-found.tsx` is the canonical handler for 404 responses (whether they come from a `notFound()` call or from an unmatched route). The two pages have different layouts and SEO implications, so a project that adds `error.tsx` without `not-found.tsx` ends up with a half-finished error story.

The build warns (rather than errors) because the default 404 fallback is functional. The warning is a nudge to complete the picture.

## Reproducing the error

```tsx
"use client";

export default function Error({ error, reset }: { error: Error; reset: () => void }) {
  return (
    <div>
      <h1>Something went wrong</h1>
      <button onClick={reset}>try again</button>
    </div>
  );
}
```

## Error output

```
[WARN] [SR0033] custom error page without a 404 page
   --> app/error.tsx:1:1
   |
 1 | "use client"
   | ^^^^^^^^^^^^^^ add app/not-found.tsx with a 404 UI to match this error page
   |
  help: a custom error page assumes a custom 404 page. Add app/not-found.tsx with the same look.
```

## How to fix it

1. Add an `app/not-found.tsx` that renders a friendly 404 message and a link back home.
2. Use the same layout as the rest of the site (the file lives in the same segment as `error.tsx`, so it inherits the surrounding layout).
3. Return a `metadata` export so the page title reads "Not Found" instead of the route slug.

```tsx
import Link from "swift-rust/link";

export const metadata = { title: "Not Found" };

export default function NotFound() {
  return (
    <div>
      <h1>Not found</h1>
      <p>The page you were looking for does not exist.</p>
      <Link href="/">go home</Link>
    </div>
  );
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The trace log lists every segment that defines `error.tsx` but no `not-found.tsx`, so you can fix all of them in one pass.

## Common pitfalls

- A `not-found.tsx` placed in a nested segment only handles 404s for that segment. The root `app/not-found.tsx` is the catch-all.
- The file must be named exactly `not-found.tsx`. `404.tsx` is the pages-router name and is not recognized here.
- `notFound()` is the canonical way to trigger a 404 from a server component. Throwing a 404 status from a route handler is the route-handler equivalent.
- This warning is unrelated to `SR0028` (public file vs page collisions), even though both involve 404s.

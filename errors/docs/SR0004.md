# SR0004 — Bundling error

Bundling runs after compilation. It copies static assets from `public/`, generates route manifests, fingerprints the client chunks, and invokes `wasm-pack` to build the WASM hydration bundle when the project uses `ssr-wasm` or `wasm` rendering.

## Why this error occurred

The bundler is the last step that touches the filesystem before the server starts. If it cannot read an asset, write into `.swift-rust/`, or invoke the toolchain needed for WASM, the build fails here — usually with a message that names the missing piece directly.

The output directory is created lazily, so a "permission denied" on a path the bundler is trying to *create* (not just write to) is also possible.

## Reproducing the error

```tsx filename="app/hero.tsx"
import { Image } from "swift-rust/image";

export default function Hero() {
  return <Image src="/hero.jpg" width={1200} height={600} priority alt="Hero" />;
}
```

```text filename="public/"
```

The `public/hero.jpg` file is missing. The bundler tries to fingerprint it during the build and fails.

```bash
bun swift-rust build --rendering wasm
# error: [SR0004] bundling error: `wasm32-unknown-unknown` target not installed
```

The project is set to `wasm` rendering but the Rust target is not installed.

## Error output

```

```

## How to fix it

1. If an asset is missing, either add the file to `public/` or fix the `src` prop on the component that references it.
2. If the output directory is read-only, change the `build.outputDir` field in `swift-rust.config.json` to a writable path, or fix the permissions on the current directory.
3. If the WASM target is missing, install it:

```bash
rustup target add wasm32-unknown-unknown
curl https://rustwasm.github.io/wasm-pack/installer/init.sh | sh
```

4. Re-run the build to confirm.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust::bundler=trace bun swift-rust build
ls -la .swift-rust/
```

The trace log walks the bundler pipeline step by step and shows which stage failed. `ls -la .swift-rust/` confirms the output directory is writable.

## Common pitfalls

- The bundler reads `public/` *and* any path passed to `Image.src`, `Font.src`, or `static()`. A typo in any of those trips this error.
- `wasm-pack` is a one-time install. Upgrading Rust does not remove it, but switching Rust toolchains (e.g. `rustup default nightly`) can move it out of `PATH` if installed in a per-toolchain prefix.
- The bundler does not delete stale files in `.swift-rust/`. If you previously built with a different rendering mode, run `bun swift-rust clean` before switching.
- A symlink loop in `public/` (a symlink that points back to itself) crashes the bundler with `too many levels of symbolic links`, not a missing-file error. Inspect `public/` for self-references if the diagnostic output looks wrong.

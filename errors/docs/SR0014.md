# SR0014 — Dynamic href is not supported in the app router

A `<Link>` from `swift-rust/link` was given an object-shaped `href` (the `pathname`/`query` form) or a partially interpolated path. The app router resolves URLs by string, not by route mapping, so dynamic shapes cannot be turned into a navigable URL at runtime.

## Why this error occurred

In the legacy pages router, `<Link>` accepted an object so it could fill in dynamic segments and append a query string. The app router ships a real URL-based router: the `href` is the URL. There is no client-side route table to look up, so an object cannot be resolved.

Strings with literal `[slug]` placeholders also fail for the same reason — the router does not know what `[slug]` is supposed to be without a concrete value.

## Reproducing the error

```tsx
import Link from "swift-rust/link";

export function Card({ id }: { id: string }) {
  return (
    <Link href={{ pathname: "/post/[slug]", query: { slug: id } }}>
      read more
    </Link>
  );
}
```

## Error output

```
[ERROR] [SR0014] dynamic href is not supported in the app router
   --> app/components/card.tsx:5:14
   |
 5 |     <Link href={{ pathname: "/post/[slug]", query: { slug: id } }}>
   |              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ pass a string URL, e.g. `/post/${id}`
   |
  help: replace the object href with a string. The app router does not interpolate path segments.
```

## How to fix it

1. Compute the final URL yourself before handing it to `<Link>`.
2. Make sure every dynamic segment is filled in with a real value, not a template literal containing `[param]`.
3. If you have many links, a tiny `href(post)` helper keeps the call sites readable.

```tsx
import Link from "swift-rust/link";

export function Card({ id }: { id: string }) {
  return <Link href={`/post/${id}`}>read more</Link>;
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

The trace log points at the exact `<Link>` JSX site and the offending `href` value, which is helpful when a deep component tree forwards an object prop.

## Common pitfalls

- The legacy pages-router shape (`{ pathname, query }`) is the most common cause — it was a major public API in earlier versions and is easy to copy-paste from old code.
- A `String(href)` cast around the object does not help; the router only checks the type, not the result.
- Forwarding `href` through a prop with `as Route` typings from a shared package can hide the error inside a wrapper. Log the value once to confirm.
- This is distinct from `SR0055` (interpolation failure) and `SR0059` (`href` + `as` mismatch).

# SR0073 — Invalid page or API route config

In Swift-Rust, an API route can opt into special runtime behaviour by exporting a `config` object:

```ts
export const config = {
  runtime: "edge",
  regions: ["iad1", "fra1"],
};
```

The framework reads this object at build time to decide how to compile the route. The contract is that `config` must be a static, in-place object literal: a plain JSON-shaped value with no expressions, no re-exports, and no computed fields.

## Why this error occurred

The build pipeline statically analyses the `config` export to determine routing decisions (which runtime, which regions, which cache). Anything that cannot be resolved at build time is rejected:

- A string (`export const config = 'hello'`) — the framework expects an object.
- A dynamic expression (`runtime: \`node${'js'}\``) — template literals defeat static analysis.
- A re-export (`export { config } from '../shared-config'`) — the source file's content is invisible to the analyser.
- A non-primitive value (`regions: getRegions()`) — function calls are not allowed.

The error fires at the next dev rebuild, or at the start of `bun swift-rust build`. It is raised against the offending line in the file that should have declared the config.

## Reproducing the error

```ts
// app/api/legacy/route.ts
import type { RequestHandler } from "swift-rust/server";

export const config = {
  runtime: `node${"js"}`,
};

export const GET: RequestHandler = async () => {
  return new Response("ok");
};
```

## Error output

```
error: [SR0073] invalid page or API route config
  --> app/api/legacy/route.ts:4:3
   |
 4 |   runtime: `node${"js"}`,
   |           ^^^^^^^^^^^^^^
   |
   = the value of `config.runtime` must be a static string literal.
     template expressions are not supported.
   = hint: use `runtime: "nodejs"` (or `"edge"`).
```

## How to fix it

1. Use a static object literal with primitive values.

   ```ts
   // app/api/legacy/route.ts
   import type { RequestHandler } from "swift-rust/server";

   export const config = {
     runtime: "nodejs",
   };

   export const GET: RequestHandler = async () => {
     return new Response("ok");
   };
   ```

2. For the edge runtime:

   ```ts
   export const config = {
     runtime: "edge",
     regions: ["iad1", "fra1"],
   };
   ```

3. If multiple routes need the same config, duplicate it in each file. Re-exporting from a shared module is not supported.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=config=trace bun swift-rust build
rg -n "export const config" app/
```

## Common pitfalls

- Template literals are the most common offender. `\`node${'js'}\`` looks like the string `'nodejs'` but is a dynamic expression.
- `regions: ['*']` is the only wildcard value. `'*'.repeat(1)` is rejected.
- `export { config } from '../x'` and `export * from '../x'` (where `x` exports a `config`) are both rejected.
- This error is the modern equivalent of Next.js's `invalid-page-config`. The contract is identical.

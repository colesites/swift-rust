# SR0072 — Invalid Swift-Rust config

The project config (`swift-rust.config.json`, `swift-rust.config.ts`, `swift-rust.config.toml`, or `swift-rust.config.yaml`) is parsed against a strict schema at boot. Any field that is unknown, deprecated, or has the wrong type aborts the build with this error. The framework deliberately fails loudly rather than silently dropping a misconfigured option, because the cost of a missed setting is usually a subtle runtime regression.

## Why this error occurred

The config is the single source of truth for build, runtime, and bundler behaviour. To keep it predictable, the framework refuses to start when it cannot make sense of the file. Common causes:

- A typo (`imag` instead of `image`).
- A deprecated or removed option (`experimental.turbo` — moved to the top-level `bundler.turbopack`; `eslint` — moved to the ESLint CLI).
- A value of the wrong type (`true` instead of `'default'` for `image.loader`).
- An experimental option that was promoted or removed in the current release.
- A hand-rolled object passed where a primitive is expected (e.g. `assetPrefix: { url: '…' }` instead of a string).

The error message points to the offending line and explains what was expected.

## Reproducing the error

```json
{
  "imag": {
    "domains": ["cdn.example.com"]
  }
}
```

## Error output

```
error: [SR0072] invalid Swift-Rust config
  --> swift-rust.config.json:2:3
   |
 2 |   "imag": {
   |   ^^^^^^^
   |
   = unknown field `imag`. did you mean `image`? expected one of:
     rendering, image, font, pdf, routing, bundler, compiler, server,
     platform, telemetry.
```

## How to fix it

1. Use the TypeScript config file for compile-time validation. The `SwiftRustConfig` type rejects unknown fields.

   ```ts
   // swift-rust.config.ts
   import type { SwiftRustConfig } from "swift-rust";

   const config: SwiftRustConfig = {
     image: {
       remotePatterns: [{ protocol: "https", hostname: "cdn.example.com" }],
     },
   };

   export default config;
   ```

2. Migrate removed options to their successors.

   ```json
   {
     "bundler": {
       "turbopack": {
         "rules": {}
       }
     }
   }
   ```

   The `eslint` option is no longer a top-level key; configure ESLint via the `eslint.config.js` file at the project root and run it as a separate CI step.

3. If the config is generated programmatically, validate it before writing it to disk.

   ```ts
   import { validateConfig } from "swift-rust/config";

   const parsed = JSON.parse(raw);
   validateConfig(parsed);
   ```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=config=trace bun swift-rust build
```

## Common pitfalls

- The `experimental.*` namespace is reserved. Options under it are renamed to a stable, top-level key when they graduate. The codemod `experimental-cleanup` rewrites known migrations.
- The `images` key was renamed to `image` in the first Swift-Rust release; old Next.js configs need the rename.
- A common migration mistake is leaving both `experimental.turbo` and `bundler.turbopack` set. The framework warns and the new key wins, but the error here would catch the typo first.
- This is the Swift-Rust analogue of Next.js's `invalid-next-config`. The behaviour matches.

# SR0021 — Caches bypassed in development

A request to the dev server was served with all server caches bypassed. The build prints a warning so you can tell whether the slow path you are seeing is real or simply the result of a forced refresh.

## Why this error occurred

The dev server tries to fill the same per-segment caches that production uses, so a warm navigation feels instant and React DevTools can highlight which parts of the tree are static. When the client signals `cache-control: no-cache` (the default in Chrome while DevTools is open) or `draftMode` is enabled, the cache fill is skipped — otherwise every request would be cold and the dev experience would be miserable.

The warning is informational. The render still completes, just without the cache warmup. In production this never fires; the bypass is a dev-only signal.

## Reproducing the error

```bash
curl -H "cache-control: no-cache" http://localhost:3000/dashboard
```

## Error output

```
[WARN] [SR0021] caches bypassed in development: cache-control: no-cache
   request: GET /dashboard
   |
   help: disable "Disable Cache" in DevTools, or exit draft mode, to restore the dev cache warmup.
```

## How to fix it

1. Open DevTools → Network and uncheck "Disable Cache" once you are done forcing a refresh.
2. If `draftMode` is on, turn it off in the page that toggled it (`/api/draft/disable` is a common pattern).
3. To deliberately invalidate a cache without bypassing all of them, call `revalidateTag("dashboard")` from a server action or webhook instead of a hard refresh.

```tsx
"use server";
import { revalidateTag } from "swift-rust/cache";

export async function refreshDashboard() {
  revalidateTag("dashboard");
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

The trace log shows the cache key that was bypassed and the header that caused it, so you can confirm the source (DevTools vs draft mode).

## Common pitfalls

- Chrome defaults to "Disable Cache" while DevTools is open. A slow page is not always a real regression.
- `draftMode` is intentional. If you are previewing unpublished content, the warning is expected.
- Hard refreshes during testing are not representative of real user behavior. Measure with caches warm.
- A persistent stream of this warning in CI usually means a test fixture is sending `cache-control: no-cache` on every request.

# SR0038 — Dynamic server usage error

A request-time API (`cookies()`, `headers()`, `connection()`) was called from a code path that is no longer bound to the request's async context. The build/runtime cannot associate the call with a request, so it raises this error rather than returning potentially-wrong data.

## Why this error occurred

Swift-Rust's server runtime uses async context (a TC39 proposal) to thread the current request through the call stack. Functions like `cookies()` read from that context. The context only lives for the duration of the synchronous call chain attached to the request. Operations like `setTimeout`, `setInterval`, event handlers, and unawaited promises create a **new** execution context, and the original request context does not survive the hop.

Reading a request-time API from inside one of those new contexts is unsafe: the framework would have to either return a stale value, throw, or invent a synthetic request. The current behavior is to throw so the bug is caught early.

## Reproducing the error

```tsx
import { cookies } from "swift-rust/headers";

async function getCookieData() {
  return new Promise((resolve) =>
    setTimeout(async () => {
      const jar = await cookies();
      resolve(jar.getAll());
    }, 1000),
  );
}

export default async function Page() {
  const data = await getCookieData();
  return <p>cookies = {data.length}</p>;
}
```

## Error output

```
[ERROR] [SR0038] dynamic server usage: cookies() called outside its async context
   --> app/page.tsx:6:22
   |
 6 |       const jar = await cookies();
   |                      ^^^^^^^^^^^^^ called from setTimeout; the request context is lost
   |
  help: read the value before the async boundary, then capture it in a local variable.
```

## How to fix it

1. Read the request-time value in the component body (or at the top of the enclosing function), where the async context is still live.
2. Capture the value into a local variable.
3. Use the local variable inside any callbacks that fire later.

```tsx
import { cookies } from "swift-rust/headers";

async function getCookieData() {
  const jar = await cookies();
  const all = jar.getAll();
  return new Promise((resolve) => setTimeout(() => resolve(all), 1000));
}

export default async function Page() {
  const data = await getCookieData();
  return <p>cookies = {data.length}</p>;
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

The trace log prints the full call chain that crossed the async boundary, so you can see exactly where the context was lost. A stack frame for the `setTimeout` callback is the giveaway.

## Common pitfalls

- An `await` inside a `forEach` callback does not propagate — the surrounding function returns before the callback runs. Use `for ... of` or `Promise.all` with `map`.
- Event handlers in client components (button onClick, etc.) run in a fresh context and have no access to request APIs. The page that owns the handler is the boundary.
- A `useEffect` on the client side also runs in a fresh context. Move the request read to a server component.
- This is the runtime sibling of `SR0015` (build-time static-to-dynamic). Fixing one usually fixes the other.

# SR0093 — Module not found

The bundler could not resolve an import. The build cannot continue without a concrete file or package behind every import statement, so it stops with a list of the candidates it tried.

## Why this error occurred

There are a handful of common causes, and the error message usually points to the right one. The bundler reports:

- The import specifier as written in the source.
- The set of candidate paths it tried on disk.
- Whether the package exists in `node_modules` and is reachable.

The most common cases are: a missing dependency, a typo in the path, a case mismatch (the bundler runs on case-sensitive filesystems by default), or a Node.js-only package being imported into code that will run in the browser or in a WASM target. Swift-Rust's compiler is strict about the last one: if you import `node:fs` from a component that ends up in a client bundle, the build fails because `fs` does not exist in the browser.

## Reproducing the error

```tsx filename="app/redis-cache.ts"
import Redis from "ioredis"

const redis = new Redis(process.env.REDIS_URL)

export async function getCachedPosts() {
  return redis.get("posts")
}
```

```tsx filename="app/page.tsx"
import { getCachedPosts } from "./redis-cache"

export default async function Page() {
  const data = await getCachedPosts()
  return <pre>{data}</pre>
}
```

`ioredis` is a Node.js-only library; importing it at the top of a module that is reachable from a client component will trip the bundler.

## Error output

```
error: [SR0093] module not found: ioredis (used from app/page.tsx)
  --> app/redis-cache.ts:1:18
   |
 1 | import Redis from "ioredis"
   |                  ^^^^^^^^
   |  ioredis imports node:net and node:stream, which are unavailable in the browser
   |  candidates:
   |    /Users/me/project/node_modules/ioredis  (present, but blocked by target compatibility)
```

## How to fix it

1. **Missing dependency.** Install it: `bun add swr`.
2. **Wrong path.** Check the import path and the file name. Casing matters on Linux and on macOS in production builds.
3. **Node.js-only import.** Move the import into a server component or server action. Server components are guaranteed to never be bundled for the client.

```tsx filename="app/posts/page.tsx"
import "server-only"
import Redis from "ioredis"

const redis = new Redis(process.env.REDIS_URL)

export default async function Page() {
  const raw = await redis.get("posts")
  const posts = JSON.parse(raw ?? "[]")
  return (
    <ul>
      {posts.map((p) => <li key={p.id}>{p.title}</li>)}
    </ul>
  )
}
```

```tsx filename="app/api/posts/route.ts"
import { db } from "@/lib/db"

export async function GET() {
  const posts = await db.post.findMany()
  return Response.json(posts)
}
```

## Diagnosis

```bash
bun swift-rust doctor --check resolution
bun swift-rust info --tree
```

`--tree` prints the dependency graph from each route, which makes it easy to see who is pulling in a forbidden module.

## Common pitfalls

- A package that uses `node:` builtins under the hood (Prisma client, bcrypt, sharp) cannot be loaded in a client component. Look for `"server-only"` in third-party package READMEs.
- Re-exporting from a barrel file (`export * from "./redis-cache"`) hides the offending import. The bundler still reports it, but at a less obvious location.
- TS path aliases (`@/lib/...`) are resolved by the bundler, not by Node. Make sure they are also defined in `tsconfig.json` `paths`.

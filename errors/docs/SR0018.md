# SR0018 — Build directory is not writeable

Swift-Rust tried to emit a file under `distDir` (default `.swift-rust/`) and the operating system refused. The build aborts because every subsequent write would also fail, and continuing would leave a half-written output on disk.

## Why this error occurred

The most common cause is a production environment that mounts the application image as read-only. The image ships a prebuilt `.swift-rust/` directory from the build stage, but the runtime starts a process that expects to be able to write caches and tracing artifacts into the same path. The OS returns `EROFS` and Swift-Rust surfaces it as this error.

A second common cause is running a custom server with `bun swift-rust dev` after a production build. The dev path tries to write HMR manifests and request logs into the same directory, which is read-only in production.

## Reproducing the error

```json
{
  "name": "site",
  "output": "standalone"
}
```

```bash
bun swift-rust build
bun swift-rust dev
```

## Error output

```
[ERROR] [SR0018] build directory is not writeable: .swift-rust
   --> swift-rust.config.json:2:14
   |
 2 |   "output": "standalone"
   |              ^^^^^^^^^^^^
   |
  detail: cannot open '.swift-rust/cache.json' for writing: Read-only file system (os error 30)
  help: mount a writable volume at .swift-rust/, or run the dev server with a different HOME/cache path.
```

## How to fix it

1. Mount a writable volume at the build directory in production. The standard Swift-Rust image exposes this as `/tmp/.swift-rust`.
2. Set `SWIFT_RUST_CACHE_DIR` to a writable path if you cannot change the mount.
3. Make sure your `package.json` scripts use the right command for the right environment (`build` for production images, `dev` for local development only).

```json
{
  "name": "site",
  "output": "standalone",
  "distDir": "/tmp/.swift-rust"
}
```

```bash
export SWIFT_RUST_CACHE_DIR=/tmp/.swift-rust
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The trace log shows the exact syscall that failed and the path it was targeting. That is usually enough to figure out which mount is read-only.

## Common pitfalls

- `distDir` defaults to `.swift-rust/`. If you forgot to mount a writable volume for it, every boot will fail.
- Docker `COPY --chown=node:node` does not by itself make the image read-only; you still need to point the runtime at a writable cache.
- A previous build as root leaves files that a non-root runtime cannot overwrite. Either run the build as the same user or wipe the directory first.
- The same error fires when the **parent** directory is not writeable, even if `distDir` itself does not yet exist.

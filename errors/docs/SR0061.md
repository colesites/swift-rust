# SR0061 — Install `sharp` to use built-in image optimization

Swift-Rust's `<Image>` component from `swift-rust/image` relies on `sharp` to resize, re-encode, and serve images at the right resolution for the requesting device. The framework does not bundle a copy of `sharp` because the underlying library is a native binding that has to be compiled for the exact platform it runs on. If the install fails, image optimization falls back to a passthrough that serves the source image unchanged, and this `info` message is printed.

## Why this error occurred

`sharp` is a Node-API binding to the libvips C library. It ships prebuilt binaries for the most common platforms (darwin-arm64, darwin-x64, linux-x64, linux-arm64, win32-x64) and falls back to building from source. The install can fail for several reasons:

- The active Node version does not match the `sharp` ABI. The `sharp` team tags binaries per Node major; a Node 18 install will not pick up a Node 20 binary.
- The OS does not have the build tools (`python`, `make`, `g++`, `libvips-dev`) needed to compile from source.
- On macOS, the XCode Command Line Tools are not installed.
- On Windows, the runtime is the standard Node, not the Windows Subsystem for Linux; the prebuilt is for Linux.
- A firewall or proxy is blocking the GitHub Releases download that hosts the prebuilt binaries.

This is `info`-level because the dev server does not crash. The downside is that production image URLs return the source image at full size, which is both slow and visually broken on responsive layouts.

## Reproducing the error

```tsx
import Image from "swift-rust/image";

export default function Avatar() {
  return <Image src="/me.png" alt="me" width={128} height={128} />;
}
```

```bash
bun swift-rust build
```

## Error output

```
info: [SR0061] install `sharp` to use built-in image optimization
  --> app/page.tsx:4:1
   |
   = the `sharp` package is not installed or failed to load. images will
     be served unchanged. run `bun add sharp` and verify your platform
     is supported.
```

## How to fix it

1. Install `sharp` (or `bun add sharp`) on a Node version that matches your platform's prebuilt binary.

   ```bash
   bun add sharp
   ```

2. On macOS, install the XCode Build Tools so the source build can succeed if the prebuilt is unavailable.

   ```bash
   xcode-select --install
   ```

3. On Windows, prefer WSL for development. Sharp's prebuilt for win32-x64 is maintained but lags behind the linux/darwin builds by a few weeks.

4. As a last resort, use the WASM build. It is much slower but always installs.

   ```bash
   bun add --cpu=wasm32 sharp
   ```

5. If you do not need image optimization, set `unoptimized: true` in the project config to silence the warning.

   ```json
   {
     "image": {
       "unoptimized": true
     }
   }
   ```

## Diagnosis

```bash
bun swift-rust doctor
ls node_modules/@img/sharp-* 2>/dev/null || echo "no sharp platform binary"
node -e "console.log(process.platform, process.arch, process.versions.node)"
RUST_LOG=swift_rust=image=trace bun swift-rust dev
```

## Common pitfalls

- The `info` severity means the build does not fail. The dev server is fully functional, but `<Image>` URLs return the source image.
- Sharp's prebuilt binaries are downloaded from `https://github.com/lovell/sharp-libvips/releases`. If your network blocks GitHub Releases, set the `SHARP_INSTALL_FORCE` env var to force a source build (and install `libvips` system-wide).
- M1 Mac users who switch from an x64 Node to an arm64 Node must reinstall `sharp`; Bun will not pick up the new ABI automatically.
- This is the Swift-Rust analogue of Next.js's `install-sharp`. The migration is identical: same package, same `unoptimized` flag, same fallback.

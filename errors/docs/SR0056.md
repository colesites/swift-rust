# SR0056 ‚Äî Ignored compiler options

Swift-Rust ships a Rust-based compiler (SWC) that handles TypeScript, TSX, and several JSX dialects. Most teams never need to configure it, but the config exposes a `compiler` block for fine-grained tweaks ‚Äî for example enabling the styled-components transformer, opting in to SWC's minifier, or stripping `console.*` calls. Those options only take effect when the Rust compiler is the one actually running. The moment a `.babelrc` is present, Swift-Rust falls back to Babel and the `compiler` block is silently dropped.

## Why this error occurred

The presence of `.babelrc`, `.babelrc.js`, or `babel.config.json` is a hard signal that the project wants Babel. The framework does not refuse to start in that mode ‚Äî many legacy codebases require it for custom plugins ‚Äî but it warns the developer that the SWC-only knobs are now dead weight.

Common reasons teams end up in this state:

- A `.babelrc` was added to support a plugin that SWC does not implement yet.
- A new developer copies a Babel config from a different project.
- The Babel config is left over from a `pages/`-era setup that was not fully migrated.
- The team enabled a SWC-specific option (e.g. `compiler.removeConsole`) expecting it to work, but the project's `.babelrc` short-circuits SWC.

## Reproducing the error

```json
// .babelrc
{
  "presets": ["swift-rust/babel"]
}
```

```json
// swift-rust.config.json
{
  "compiler": {
    "removeConsole": {
      "exclude": ["error"]
    }
  }
}
```

## Error output

```
warning: [SR0056] ignored compiler options
  --> swift-rust.config.json:2:3
   |
 2 |   "compiler": {
 3 |     "removeConsole": { "exclude": ["error"] }
 4 |   }
   |   ^^^^^^^^^^^^^
   |
   = these options only apply to the Rust-based compiler (SWC).
   = a Babel config was detected at the project root, so SWC is disabled
     and these keys are ignored.
```

## How to fix it

1. If you do not actually need Babel, delete the Babel config. The `compiler` block will take effect.

   ```bash
   rm .babelrc babel.config.json
   ```

2. If you need Babel (e.g. for a plugin SWC does not implement), drop the `compiler` block and replicate the behavior in Babel.

   ```json
   {
     "presets": ["swift-rust/babel"],
     "plugins": ["transform-remove-console"]
   }
   ```

3. If you want the build to fail loudly in this case, set the strict mode in the project config.

   ```json
   {
     "compiler": {
       "strict": true
     }
   }
   ```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=compiler=trace bun swift-rust build
ls .babelrc babel.config.json 2>/dev/null
```

## Common pitfalls

- Babel is invoked through the legacy `swift-rust/babel` preset. That preset delegates most work to SWC under the hood, so this warning is a useful indicator that the project is paying Babel's startup cost for no gain.
- The `compiler` block is parsed but never applied. Tools that statically read the config (e.g. a linter plugin) cannot know that the values are dead.
- The warning fires on every build. Combine it with `strict: true` once you have cleaned up the Babel files, so future regressions are caught.
- This used to be a hard error in older Next.js. Swift-Rust demotes it to a `warning` because many large codebases still depend on Babel plugins.

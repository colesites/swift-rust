# SR0058 — Improper devtool configuration

The Swift-Rust bundler (Rspack under the hood, with a webpack-compatible config surface) picks a sensible `devtool` value for each mode: `eval` (or `eval-source-map` in some setups) for development, and `source-map` for production. The `devtool` controls how source maps are generated, and the wrong choice can slow down rebuilds by an order of magnitude. If you set `devtool` to a slow value in the project config without scoping it to production, the dev server prints this warning.

## Why this error occurred

In development the bundler recompiles files on every save. The default `devtool: 'eval'` is fast because the source map is generated by wrapping each module in `eval("...")` and decoding the inline comment at the end. Switch to `'source-map'` (or anything that emits a real `.map` file) and every recompile has to encode a full base64 VLQ map. The first compile gets 5–10x slower, and HMR updates 10–20x slower, with no visible change to the developer.

Common causes:

- A developer pastes a config snippet from a production debugging guide without checking the `mode` flag.
- The team wants real source maps in dev for use with a debugger extension; in that case use `'eval-cheap-module-source-map'`, which is much faster than `'source-map'` and still gives line-accurate (but not column-accurate) maps.
- The `devtool` override is applied via a webpack `config.devtool = '...'` line that is not gated on `options.dev`.

The warning is non-fatal: the build still succeeds, but the developer experience is degraded.

## Reproducing the error

```ts
// swift-rust.config.ts
import type { SwiftRustConfig } from "swift-rust";

const config: SwiftRustConfig = {
  bundler: (config, options) => {
    config.devtool = "source-map";
    return config;
  },
};

export default config;
```

## Error output

```
warning: [SR0058] improper devtool in development
  --> swift-rust.config.ts:5:5
   |
 5 |     config.devtool = "source-map";
   |     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
   |
   = devtool "source-map" can slow dev rebuilds by 10x. use "eval" or
     "eval-cheap-module-source-map" in development. scope the override
     to production with `if (!options.dev)`.
```

## How to fix it

1. Gate the devtool override on the production mode.

   ```ts
   import type { SwiftRustConfig } from "swift-rust";

   const config: SwiftRustConfig = {
     bundler: (config, options) => {
       if (!options.dev) {
         config.devtool = options.isServer ? false : "source-map";
       }
       return config;
     },
   };

   export default config;
   ```

2. Or remove the override entirely and let the framework pick.

3. If you need readable stack traces during local debugging, prefer `'eval-cheap-module-source-map'`, which is still very fast.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=bundler=trace bun swift-rust dev
```

## Common pitfalls

- The warning fires only when the devtool is set in development. The same value in production is fine.
- `false` is a valid devtool value (it disables source maps entirely). It is the fastest option but gives no stack traces — usually only acceptable for the server build.
- The framework's default in dev is `eval` (fastest, no separate file). The default in production is `source-map` (one external `.map` per chunk).
- This is a `warning`, not an `error`. The build does not fail. Treat the message as a perf-budget nudge.

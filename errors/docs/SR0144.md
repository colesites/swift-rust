# SR0144 — Static directory is deprecated

Swift-Rust (and its predecessor) used to serve static assets from a top-level `static/` directory. That directory is deprecated; the canonical location is now `public/`. The build emits a deprecation warning the first time it sees a `static/` directory and continues to serve from it, but the directory will be removed in a future major release.

## Why this error occurred

The `public/` directory was introduced so that assets could be served from the root of the site (e.g. `/hero.png`) without an explicit prefix. Keeping both `static/` and `public/` doubled the surface area for routing rules and confused users about which directory an asset should go in.

The deprecation is enforced as a warning today. The build succeeds, and assets from `static/` are still served. In the next major version, the directory will be ignored entirely, so removing it now avoids a silent breakage.

## Reproducing the error

```
project/
├── static/
│   └── hero.png
├── app/
│   └── page.tsx
└── swift-rust.config.json
```

## Error output

```
[WARN] [SR0144] static directory is deprecated
  --> static/
    |
    | `static/` is deprecated; move its contents to `public/` and remove this directory.
    | assets currently served from `/<file>` will continue to work until the next major release.
    |
    help: `mv static/* public/` and update any import paths.
```

## How to fix it

1. Move the contents of `static/` into `public/`:

```bash
mv static/* public/
```

2. Update any code that imports from `static/`:

```tsx
// before
import hero from "../static/hero.png";

// after
import hero from "../public/hero.png";
```

3. Remove the now-empty `static/` directory:

```bash
rmdir static
```

4. Verify the build still serves the assets:

```bash
curl -I http://localhost:3000/hero.png
```

## Diagnosis

```bash
ls static public
bun swift-rust doctor
```

The doctor output lists the resolved `publicDir` and flags a legacy `static/` directory if it is present.

## Common pitfalls

- The two directories are not merged. A file in `static/foo.png` is served from `/foo.png`, and a file in `public/foo.png` is also served from `/foo.png`. The two would shadow each other; the framework picks one and warns.
- A Vite or CRA template may still scaffold a `static/` directory. Remove it on import.
- A monorepo with shared assets in `packages/ui/static/` is fine — only the top-level `static/` triggers the warning.
- The deprecation is silent in some bundler output modes. Pass `--verbose` to make sure the warning is printed.

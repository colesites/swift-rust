# SR0006 — Image error

The image service is invoked both at build time (for `priority` images) and at request time (for `/_swift-rust/image`). If the source is unreachable, in an unsupported format, or hosted on a domain that is not allow-listed, the service raises this error.

## Why this error occurred

The `<Image>` component does two things: it resizes and re-encodes the source at build time (or on first request) and it serves the optimized bytes from the cache. Both steps can fail. The build-time path runs once per `priority` image; the request-time path runs on first hit and is then cached.

The framework refuses to fetch from a domain that is not allow-listed because the image proxy is otherwise a free SSRF vector. Adding a domain is a deliberate act, not an implicit one.

## Reproducing the error

```tsx filename="app/hero.tsx"
import { Image } from "swift-rust/image";

export default function Hero() {
  return (
    <Image
      src="https://evil.example.com/x.jpg"
      width={1200}
      height={600}
      priority
      alt="Hero"
    />
  );
}
```

`evil.example.com` is not in `image.domains`. The service refuses to fetch.

```tsx filename="app/hero.tsx"
<Image src="/hero.bmp" width={1200} height={600} priority alt="Hero" />
```

BMP is not in the supported format list. The service rejects the source.

## How to fix it

1. Add the domain to your config (deliberate, not implicit):

```json filename="swift-rust.config.json"
{
  "image": {
    "domains": ["cdn.example.com", "evil.example.com"],
    "formats": ["avif", "webp"],
    "qualities": [75, 85]
  }
}
```

2. Convert the source to a supported format (`bunx sharp -i hero.bmp -o hero.webp` or similar).
3. Re-run the build or hit the route again to populate the cache.

```tsx filename="app/hero.tsx"
<Image src="/hero.webp" width={1200} height={600} priority alt="Hero" />
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info --images
RUST_LOG=swift_rust::image=trace bun swift-rust dev
```

`info --images` lists every image the framework knows about, with the resolved URL and cache state. The trace log prints the allow-list check, the fetch attempt, and the cache lookup.

## Common pitfalls

- The allow-list is a separate config from the route's `headers.host`. A CDN domain that serves images may be different from the CDN domain the site is hosted on.
- The first request to a remote image is slow (the fetch + optimize + cache) and the failure happens in that first request. Subsequent requests are served from cache. Use `priority` to surface the failure at build time instead.
- The cache is per-host, so the same image at two different `image.domains` entries will be optimized twice.
- Forgetting to install `sharp` in production is a deployment-time failure, not a build-time failure. The dev server uses the local `sharp` from `node_modules`, but the production binary links against a pre-built `sharp` artifact.

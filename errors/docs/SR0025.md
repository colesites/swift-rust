# SR0025 — Class component in server component

A server component tried to render a class component. The server tree only supports function components because the rendering model is async-first; class lifecycle methods do not fit the streaming protocol.

## Why this error occurred

Server components run inside the Swift-Rust server runtime, which uses concurrent rendering and an async data model. Class components rely on synchronous `render` and `setState` semantics that are at odds with the streaming RSC payload — instance state cannot be resumed on a different worker, and the `forceUpdate` path has no client to bounce to.

Function components, by contrast, are pure functions of their props and fit the model cleanly. The compiler enforces this by rejecting any class component in the server tree.

## Reproducing the error

```tsx
import React from "react";

export default class Page extends React.Component {
  render() {
    return <p>hello</p>;
  }
}
```

## Error output

```
[ERROR] [SR0025] class component in server component: server components must be functions
   --> app/page.tsx:3:1
   |
 3 | export default class Page extends React.Component {
   | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ server components only support function components
   |
  help: convert the class to a function, or add 'use client' to this file.
```

## How to fix it

1. Convert the class component to a function component. The body of `render()` becomes the function body.
2. If you depend on instance state, lift it to a server cache (`use cache`) or to a client component.
3. If you cannot rewrite the class (e.g. it comes from a third-party library), add `'use client'` at the top of the file that owns the class.

```tsx
export default function Page() {
  return <p>hello</p>;
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The trace log lists every class component it encountered in the server tree, so you can confirm the boundary is the one you intended.

## Common pitfalls

- A class component in a deeply nested import is easy to miss. The trace log's stack trace points to the import site.
- A library that exports a class component but does not mark itself as a client component is a library bug, not yours. File an issue.
- Some bundlers expose class components in their type definitions even when the runtime is a function. The build still checks at runtime.
- This error is related to but distinct from `SR0029` (`createContext` in a server component).

# SR0077 — Invalid bundler `resolve.alias`

The bundler (Rspack, with a webpack-compatible config surface) uses `resolve.alias` to map module specifiers to on-disk paths. Swift-Rust registers a set of internal aliases — `swift-rust/link`, `swift-rust/image`, `swift-rust/font`, the special `private/swc` helpers, and others — that the framework code itself imports. If a project config reassigns `resolve.alias` without preserving the framework's entries, the imports resolve to `undefined` and the bundler throws this error at the first import.

## Why this error occurred

This is a foot-gun, not a framework bug. JavaScript objects are reference types, so:

```ts
config.resolve.alias = {
  "@/components": path.resolve(__dirname, "src/components"),
};
```

…replaces the alias map entirely, dropping every entry the framework registered. The bundler then tries to resolve `swift-rust/link` and fails.

The fix is to spread the existing map:

```ts
config.resolve.alias = {
  ...config.resolve.alias,
  "@/components": path.resolve(__dirname, "src/components"),
};
```

The error fires at the first compile that hits an aliased import — usually the dev server's initial build.

## Reproducing the error

```ts
// swift-rust.config.ts
import type { SwiftRustConfig } from "swift-rust";
import path from "node:path";

const config: SwiftRustConfig = {
  bundler(config) {
    config.resolve.alias = {
      "@/components": path.resolve(__dirname, "src/components"),
    };
    return config;
  },
};

export default config;
```

```tsx
import Link from "swift-rust/link";

export default function Page() {
  return <Link href="/about">About</Link>;
}
```

## Error output

```
error: [SR0077] invalid bundler resolve.alias
  --> app/page.tsx:1:1
   |
 1 | import Link from "swift-rust/link";
   |                       ^^^^^^^^^^^^
   |
   = cannot resolve "swift-rust/link". the project config reassigned
     resolve.alias and removed the framework's internal aliases. spread
     the existing aliases before adding new ones.
```

## How to fix it

1. Spread the existing aliases before adding new ones.

   ```ts
   // swift-rust.config.ts
   import type { SwiftRustConfig } from "swift-rust";
   import path from "node:path";

   const config: SwiftRustConfig = {
     bundler(config) {
       config.resolve.alias = {
         ...config.resolve.alias,
         "@/components": path.resolve(__dirname, "src/components"),
       };
       return config;
     },
   };

   export default config;
   ```

2. If you use the array form of `resolve.alias`, append rather than replace.

   ```ts
   config.resolve.alias = [
     ...(config.resolve.alias as Array<unknown>),
     { name: "@/components", alias: path.resolve(__dirname, "src/components") },
   ];
   ```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=bundler=trace bun swift-rust build
rg -n "resolve.alias" swift-rust.config.ts
```

## Common pitfalls

- The error message points at the first import that failed, not at the config file. If you have many `swift-rust/*` imports, the first one is the symptom; the cause is in the config.
- The framework's internal aliases are not documented because they are not a public API. Treat the spread as mandatory even if you do not see them in the type signature.
- If you are migrating from a custom webpack config, the same gotcha applies — but the alias map is exposed under a different name in some plugins. Always spread.
- This is the Swift-Rust analogue of Next.js's `invalid-resolve-alias`. The fix is identical.

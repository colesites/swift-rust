# SR0120 — Opening a new issue

This is an informational message, not a fault. When you run `bun swift-rust info` (or any command that prints the framework version) and the installed version is behind the canary release, Swift-Rust prints a reminder to reproduce against the canary before opening a GitHub issue.

## Why this error occurred

The Swift-Rust project ships a daily canary build that contains every fix and feature that has not yet been promoted to `latest`. Some bugs are fixed in the canary the same day they are reported, and the canary check exists to save both you and the maintainers time.

The CLI performs the check passively — it does not block your command. It prints the message, prints the version comparison, and continues. The message is a courtesy, not a precondition.

## Reproducing the error

This message is triggered by an outdated install rather than by code. To reproduce:

```bash
bun swift-rust info
```

If the resolved version is older than the newest canary, the message appears below the version table.

## Error output

```
[INFO] [SR0120] newer canary available
  current:  swift-rust@0.4.7
  canary:   swift-rust@0.5.0-canary.42
  
  some issues may already be fixed in the canary release. please verify
  against canary before opening a new issue:

      bun add swift-rust@canary
  
  if the issue still reproduces, please file a report with `swift-rust doctor`
  output attached.
```

## How to fix it

1. Install the canary version:

```bash
bun add swift-rust@canary
```

2. Re-run the failing command and the reproduction steps.

3. If the issue is gone, you are done. If it is still there, gather the diagnostic bundle and open an issue:

```bash
bun swift-rust doctor --output=doctor.json
```

4. Open a new issue at <https://github.com/swift-rust/swift-rust/issues/new> and attach the JSON.

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

`doctor` writes a self-contained JSON with the Bun version, Rust toolchain, OS, environment variables (with secrets redacted), and the resolved config. Attach it to the issue to skip the back-and-forth.

## Common pitfalls

- Canary builds can contain regressions. Do not run the canary in production unless you have a rollback plan.
- Lockfiles pinned to a specific canary tag (for example `0.5.0-canary.42`) will not auto-upgrade. Re-resolve the lockfile to follow the rolling canary.
- Some private registries do not mirror the canary channel. If `bun add swift-rust@canary` fails, check the registry config.
- This message is a CLI hint and never appears in the browser or in the server log — it is only visible on the terminal where you ran the command.

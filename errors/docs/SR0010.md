# SR0010 — Unknown error

This is a catch-all for internal failures that the framework did not anticipate. If you see this, it indicates either a bug in the framework itself or a mismatch between the bundled Rust binary, the SWC plugin, and the JS/TS packages.

## Why this error occurred

The framework is built on several moving parts: a Rust HTTP server, a Rust code generator, a SWC plugin, and a TypeScript runtime. Each part is versioned independently, and the contract between them is enforced at process start. When the contract is broken, the resulting panic has no specific error code and falls through to this one.

The most common root cause is a version mismatch — e.g. running the dev server from `bun swift-rust@0.1.0` against a project that has `swift-rust-build@0.2.0-canary` installed. The contract check at startup should catch this, but a few edge cases slip through.

## Reproducing the error

This error is not reproducible from user code. It surfaces when:

- A code path the framework thought was unreachable is actually reachable (a new TSX pattern the SWC plugin did not account for).
- A third-party library panicked in a way the framework did not wrap in a `catch_unwind`.
- The bundled WASM runtime is older than the JS that loaded it.

```bash
bun swift-rust dev
# error: [SR0010] unknown error: internal invariant violated: route manifest missing entry for /blog
```


## How to fix it

1. Capture the full stack trace from the log.
2. Run `bun swift-rust info` and include the output. It lists every version of every package and binary.
3. Open an issue at https://github.com/swift-rust/swift-rust/issues with the trace, the `info` output, and a minimal reproduction.

There is no user-side fix for an internal invariant violation. The framework has a bug. Filing an issue is the right action.

## Diagnosis

```bash
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

`info` is the most useful piece — it dumps every version and config that could be involved. The trace log prints the full panic backtrace with file and line numbers, which is what the maintainers will ask for.

## Common pitfalls

- A version mismatch between the CLI and the SWC plugin is the most common cause. Pin all `swift-rust-*` packages to the same version in `package.json` and `Cargo.lock`.
- A 0.0.x or pre-release version (e.g. `0.1.0-canary.3`) may be incompatible with the current `0.1.0` stable. Run `bun pm ls swift-rust` to confirm.
- A user-space `panic!` from a third-party library is reported with the same code. The framework cannot tell the difference between "our panic" and "their panic", so the resolution path is the same: file an issue, and we'll triage.
- If the error is repeatable, capture a minimal reproduction. A `git diff` against the last working commit is the fastest way to find the trigger.

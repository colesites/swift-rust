# SR0009 — I/O error

The framework reads project source files, writes build outputs to `.swift-rust/`, and maintains caches for the image, font, and route manifest subsystems. An I/O error in any of those operations surfaces here with the underlying `os error` code from the operating system.

## Why this error occurred

The framework is a file-heavy tool. The build pipeline touches hundreds of files (source, generated Rust, intermediate caches, output bundles) and a single read or write failure anywhere in that graph aborts the build. The OS error code in the message is the most useful piece — `os error 13` is permission denied, `os error 2` is file not found, `os error 28` is no space left on device.

The most common triggers are: a `.swift-rust/` directory owned by a different user (e.g. created as `root` from a Docker build), a symlink in `public/` that points to a file that was moved or deleted, and CI runners that fill their ephemeral disk.

## Reproducing the error

```bash
sudo bun swift-rust dev
# then later, as a non-root user
bun swift-rust dev
# error: [SR0009] IO error: permission denied (os error 13) opening `.swift-rust/manifest.json`
```

A `public/hero.jpg` symlink that points to a file that no longer exists:

```bash
ln -s /tmp/missing.jpg public/hero.jpg
bun swift-rust build
# error: [SR0009] IO error: no such file or directory (os error 2) opening `public/hero.jpg`
```

A disk that ran out of space:

```bash
bun swift-rust build
# error: [SR0009] IO error: no space left on device (os error 28) writing `.swift-rust/cache/route-manifest.json`
```

## Error output

```
[ERROR] [SR0009] IO error: permission denied (os error 13) opening `.swift-rust/manifest.json`
  --> .swift-rust/manifest.json
   |
   help: change the owner of `.swift-rust/` to the current user:
         chown -R $(whoami) .swift-rust/
```

```
[ERROR] [SR0009] IO error: no space left on device (os error 28) writing `.swift-rust/cache/route-manifest.json`
  --> .swift-rust/cache/route-manifest.json
```

## How to fix it

1. Identify the OS error from the message.
2. For `os error 13` (permission denied), change the owner of the offending path:

```bash
chown -R $(whoami) .swift-rust/
```

3. For `os error 2` (no such file), inspect the path with `ls -la` and `readlink -f` to find where the symlink chain breaks.
4. For `os error 28` (no space), free up disk space or change the `build.outputDir` to a volume with more room.
5. Re-run the command to confirm.

## Diagnosis

```bash
bun swift-rust doctor
ls -la .swift-rust/
df -h .
readlink -f public/  # find broken symlinks
```

`doctor` walks the project and prints every file that the framework expects but cannot read. `readlink -f` resolves a path to its ultimate target, exposing broken symlinks.

## Common pitfalls

- A symlink loop (a symlink that points back to itself or to a parent that loops back) crashes with `os error 40` (too many levels of symbolic links). Use `find . -follow -maxdepth 10` to detect loops.
- macOS and Linux report the same error codes for the same conditions, but the message strings differ. Match on the numeric code, not the text.
- The framework does not retry on `os error 24` (too many open files). Raise the `ulimit -n` if you are opening thousands of files (e.g. a large `public/` directory).
- Windows file locking (`os error 32`) is the most common cause of "I/O error" on Windows during `bun swift-rust dev`. Close any program that has the `.swift-rust/` directory open (VS Code, an editor, an Explorer window).

# SR0106 — No build cache detected

A `bun swift-rust build` ran in a CI environment, and the framework could not find a persisted build cache. The build still succeeds, but every run redoes work that could have been shared: client bundle compilation, RSC payload generation, and the precomputed route table.

## Why this error occurred

Swift-Rust stores its build artifacts in a deterministic location keyed by the contents of your `package.json`, `bun.lock`, and the relevant source directories. In a local dev environment, this directory is on disk and persists between runs, so a rebuild is incremental: only the files that changed are recompiled.

In CI, the runner is usually ephemeral. If the cache directory is not explicitly preserved, it is wiped at the end of every job, and the next job starts from scratch. The framework detects this pattern (no cache directory on disk, or a cache directory that does not match the current lockfile) and prints this warning. It is informational only — the build proceeds — but it explains why your CI times are higher than your local builds.

## Reproducing the error

A minimal CI configuration that does not cache the build directory:

```yaml filename=".github/workflows/build.yml"
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: bun install
      - run: bun swift-rust build
      - run: bun swift-rust test
```

Every run starts from an empty cache.

## Error output

```
warn: [SR0106] no build cache detected
       CI builds will be slower and client bundles will be re-generated every run
       configure cache persistence for node_modules/.cache/swift-rust
```

## How to fix it

1. Add a cache step in CI for the Swift-Rust build directory.
2. Key the cache on the lockfile and a hash of `swift-rust.config.json` so a config change invalidates the cache automatically.
3. Restore the cache before installing dependencies; save it after the build.

```yaml filename=".github/workflows/build.yml"
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/cache@v4
        with:
          path: |
            node_modules/.cache/swift-rust
            .swift-rust
          key: ${{ runner.os }}-swift-rust-${{ hashFiles('bun.lock', 'swift-rust.config.json') }}
          restore-keys: |
            ${{ runner.os }}-swift-rust-
      - run: bun install
      - run: bun swift-rust build
      - run: bun swift-rust test
```

## Diagnosis

```bash
bun swift-rust doctor --check cache
RUST_LOG=swift_rust=cache=trace bun swift-rust build
```

`--check cache` reports the size and age of the local cache and prints the expected path.

## Common pitfalls

- Caching `node_modules` is not enough — the Swift-Rust build output lives in a subdirectory that is not in the standard Bun cache path.
- The first build in a new environment always misses the cache. The warning is expected on the first run; the fix is to ensure subsequent runs hit it.
- The cache key should include `bun.lock` (so a dependency change invalidates the cache) and the config file (so a config change invalidates it). Do not key on a commit SHA — that defeats the point of caching across runs.
- Some hosted runners disable disk caching by default for security. Check the runner's documentation for an opt-in.

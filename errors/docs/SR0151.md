## Why this error occurred

A blur placeholder is part of the <Image> contract, not an optional nicety. Without it, the user sees a pop-in as the high-resolution image loads. The component is intentionally stricter than Next.js: there is no "empty" mode. The runtime throws `ImageMissingBlurError` (with `code = "SR0151"`) at render time so the gap surfaces during development, not in production.

## Reproducing the error

```tsx
<Image src="/photo.jpg" width={1200} height={800} alt="A photo" />
```

Running this in `swift-rust dev` prints:

```
ImageMissingBlurError: The <Image> component requires `placeholder="blur"`.
Pass `placeholder="blur"` and a base64 `blurDataURL` for every <Image>.

received: undefined
```

The error is thrown at render time. SSR also throws, so a request that hits the page will fail.

## Error output

```
ImageMissingBlurError [SR0151]: The <Image> component requires `placeholder="blur"`.
    at <Image> (packages/image/src/index.ts:31:3)
    at Page (app/page.tsx:3:7)
```

The thrown error class is `ImageMissingBlurError` (importable from `@swift-rust/image`). It carries:

- `code: "SR0151"`
- `name: "ImageMissingBlurError"`
- `message` — one of three reasons: missing `placeholder`, missing `blurDataURL`, or non-data-URL `blurDataURL`.

## How to fix it

1. **Add `placeholder="blur"` and `blurDataURL` to the call site.**

   ```tsx
   <Image
     src="/photo.jpg"
     width={1200}
     height={800}
     alt="A photo"
     placeholder="blur"
     blurDataURL="data:image/jpeg;base64,/9j/4AAQ..."
   />
   ```

2. **Generate the `blurDataURL`.**

   - For static imports, prefer a build-time generator. We are working on a `swift-rust image-gen` tool; until then, the public APIs `plaiceholder`, `@unpic/placeholder`, and `blurhash` are the de-facto choices.
   - For remote images, fetch a 20x30 JPEG of the source at low quality and base64-encode it. The framework's `/_swift-rust/image` route already supports a `?blur=1` query parameter that returns such an image.
   - For placeholder color only, a 1x1 transparent PNG is a valid (if ugly) `blurDataURL`: `data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAAAAEAAAABCAQAAAC1HAwCAAAAC0lEQVR42mNkYAAAAAYAAjCB0C8AAAAASUVORK5CYII=`.

3. **Update your call sites in bulk.** The new API is a breaking change from the previous `"blur" | "empty"` placeholder. TypeScript will report every site at build time. A codemod is included in the `swift-rust migrate` command.

## Diagnosis

| Cause | Distinguishing signal |
|---|---|
| Missing `placeholder` | `received: undefined` |
| Missing `blurDataURL` | `received: undefined` (the second of the three reasons) |
| `blurDataURL` is a plain URL | `received prefix: https://...` |

The error's `message` distinguishes the three; you do not need to inspect anything else.

## Common pitfalls

- **Copy-pasting an example with `placeholder="empty"`.** The `"empty"` literal no longer typechecks. TypeScript will reject the call at build time.
- **Reusing a `blurDataURL` across many images.** A blurred thumbnail of *one* image is not a good placeholder for *another*. Either per-image encode or accept the (visible) blurriness.
- **Forgetting to set it on a `next`-style static import.** A local image imported with `import hero from "./hero.jpg"` still needs `placeholder="blur"` and a `blurDataURL` at the call site.

## Why this error occurred

PDF.js renders each page to a `<canvas>` using the browser's 2D context. Three things can fail:

1. The PDF itself contains data the renderer cannot handle (an exotic font, a broken XObject).
2. The browser refuses to allocate a large enough canvas. Most browsers cap canvas dimensions at around 16 384 × 16 384 pixels. A 4K scan at scale 4.0 can hit that limit.
3. The previous render task was cancelled because the user navigated to a different page. The cancellation is reported as a render error, but it is not a failure — it is a deliberate teardown.

The viewer distinguishes these in the `PdfError.kind` field and in the `onError` payload. A `kind: "render"` with the same `page` number that was just loaded is a real failure; a `kind: "render"` immediately after a `setPage` call is a cancellation and is safe to ignore.

## Reproducing the error

```tsx
<Pdf src="/big.pdf" initialScale={4} />
```

If `/big.pdf` contains a high-resolution scan, the viewer may fail to rasterize at 4× scale. The error UI appears, and `onError` fires with:

```
PdfError: render
  code: "SR0153"
  kind: "render"
  url: "/big.pdf"
  cause: RangeError: Canvas dimensions exceeds the limit
```

## Error output

```
PdfError: render
  code: "SR0153"
  kind: "render"
  url: "/big.pdf"
  cause: <Error from PDF.js render task>
```

## How to fix it

1. **Lower the scale.** Use `"fit-width"` (the default) or pass a number between 0.5 and 2 for typical document sizes.

2. **Pre-process the PDF.** If the source contains a broken font, re-export it from a PDF editor. PDF.js can render the vast majority of PDFs produced by modern tools.

3. **Filter render-error noise from cancellations.** In `onError`, ignore the error if it arrives immediately after a page change:

   ```ts
   const lastPageChange = useRef(0);
   <Pdf
     onPageChange={(p) => (lastPageChange.current = Date.now())}
     onError={(err) => {
       if (err.kind === "render" && Date.now() - lastPageChange.current < 100) return;
       reportError(err);
     }}
   />
   ```

4. **Use `renderTextLayer={false}` if you do not need text selection.** The text layer is a second render pass; skipping it halves the memory pressure for text-heavy pages.

## Diagnosis

| Symptom | Likely cause |
|---|---|
| Fails on page 1 of a known-good PDF | Canvas size limit; lower `initialScale`. |
| Fails only on a specific page | The page contains malformed content; re-export. |
| Fails immediately after every page change | Cancellation; filter in `onError`. |
| Fails on every page of a large PDF | Out of memory; lower the scale, split the PDF. |

## Common pitfalls

- **Rendering at scale > 2 on a 4K scan.** Almost always hits the canvas size limit. Default to `"fit-width"`.
- **Calling `onError` for cancellations.** They are not failures. Either filter them in your handler or compare against the last `onPageChange` timestamp.
- **Re-trying the same render in a loop.** If the underlying cause is a canvas size limit, retrying will not help. Reset the scale to 1.0 on error and let the user re-zoom.

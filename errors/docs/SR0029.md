# SR0029 — createContext in server component

A server component called `createContext`. Contexts require a live client tree and a provider that re-renders consumers, neither of which exists in the server tree. The build aborts because there is no way to surface the context value to consumers that run on a different machine.

## Why this error occurred

`createContext` returns an object whose provider component holds state in memory and notifies consumers when the value changes. On the server, the tree is rendered to a streaming payload and then thrown away. The provider has no opportunity to push updates because the consumers are not co-located with it. The result would be a context value frozen at the time the server tree was emitted, with no mechanism for the client to receive updates — which is not a useful contract.

The fix is to make the context module a client component so the provider lives in the browser.

## Reproducing the error

```tsx
import { createContext } from "react";

const ThemeContext = createContext("light");
```

## Error output

```
[ERROR] [SR0029] createContext in server component
   --> app/theme.ts:3:1
   |
 3 | const ThemeContext = createContext("light");
   |                  ^^^^^^^^^^^^ createContext only works in client components
   |
  help: add 'use client' at the top of this file, or move the context definition to a client module.
```

## How to fix it

1. Add `'use client'` at the top of the file that calls `createContext`.
2. Keep the provider component in the same file (it is also client-only).
3. Import the context from a server component only to read it inside a client child — never to call `createContext` itself.

```tsx
"use client";
import { createContext } from "react";

export const ThemeContext = createContext("light");
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The trace log shows the offending `createContext` call site and the import path that brought it into the server tree.

## Common pitfalls

- A library that exports a context from a non-`"use client"` file will cause this error in every consumer. The fix has to come from the library.
- Re-exporting a context from a server module is also flagged, even if you do not call `createContext` yourself.
- A `useContext` call inside a server component is a different error (`SR0025` family) because the hook itself is forbidden there.
- Make sure the provider is rendered inside the client tree; an outer server `ThemeContext.Provider` will not work even if the import is allowed.

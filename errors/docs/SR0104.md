# SR0104 — Async client component

A file with the `"use client"` directive at the top exported a function declared with the `async` keyword. Client components cannot be async functions in React 18/19; the build refuses to compile this combination.

## Why this error occurred

Client components render in the browser. An async component would return a `Promise<JSX.Element>` instead of a `JSX.Element`, and React's client renderer cannot `await` a component function during render — the renderer is synchronous and is responsible for committing to a real DOM. Promises are supported in a few narrow APIs (the `use()` hook, Suspense boundaries), but the component function itself must be a regular function that returns JSX.

This used to be a hotly requested feature, and there is a [React RFC](https://github.com/acdlite/rfcs/blob/first-class-promises/text/0000-first-class-support-for-promises.md#why-cant-client-components-be-async-functions) explaining why it is not supported. The short version: making client components async would require the renderer to suspend on every component, which would cascade into a much larger redesign of the client model.

Server components, on the other hand, are async by design. The framework awaits them at request time, and there is no client-side renderer to break. If you can move the data fetching back to the server, do that.

## Reproducing the error

```tsx filename="app/posts.tsx"
"use client"

export async function Posts() {
  const res = await fetch("https://api.example.com/posts")
  const posts = await res.json()
  return (
    <ul>
      {posts.map((p) => <li key={p.id}>{p.title}</li>)}
    </ul>
  )
}
```

`Posts` is both a client component (because of `"use client"`) and async (because of the `async` keyword). The build refuses.

## Error output

```
error: [SR0104] async client component: `Posts` is declared `async` in a "use client" file
  --> app/posts.tsx:3:1
   |

   |        ^^^^^ client components cannot be async functions
```

## How to fix it

Pick one of three patterns depending on your needs.

**1. Convert to a server component.** Drop `"use client"` and the framework will await the component on the server.

```tsx filename="app/posts.tsx"
export async function Posts() {
  const res = await fetch("https://api.example.com/posts")
  const posts = await res.json()
  return (
    <ul>
      {posts.map((p) => <li key={p.id}>{p.title}</li>)}
    </ul>
  )
}
```

**2. Use `use()` with a Context Provider.** Keep the component client, but resolve the promise at the boundary.

```tsx filename="app/posts-context.tsx"
"use client"

import { createContext, use, useContext } from "react"

const PostsContext = createContext<Promise<any[]> | null>(null)

export function PostsProvider({ children, postsPromise }: { children: React.ReactNode; postsPromise: Promise<any[]> }) {
  return <PostsContext.Provider value={postsPromise}>{children}</PostsContext.Provider>
}

export function usePosts() {
  const promise = useContext(PostsContext)
  if (!promise) throw new Error("usePosts must be used within a PostsProvider")
  return use(promise)
}
```

**3. Use `useEffect` (last resort).** The data is fetched after hydration, so the user sees a loading state first.

```tsx filename="app/posts.tsx"
"use client"

import { useEffect, useState } from "react"

export function Posts() {
  const [posts, setPosts] = useState<any[] | null>(null)
  useEffect(() => {
    fetch("https://api.example.com/posts")
      .then((r) => r.json())
      .then(setPosts)
  }, [])
  if (!posts) return <p>Loading…</p>
  return (
    <ul>
      {posts.map((p) => <li key={p.id}>{p.title}</li>)}
    </ul>
  )
}
```

## Diagnosis

```bash
bun swift-rust doctor --check client-components
RUST_LOG=swift_rust=client=trace bun swift-rust build
```

`--check client-components` lists every file with `"use client"` that also has an `async` export.

## Common pitfalls

- The error fires on the `async` keyword, not on `await` inside the function. A function that awaits something synchronously is fine; an async function is not.
- "use client" applies to the file. A single `async` export in a file makes the whole file a violation, even if you intended the other exports to be server-only.
- Server actions in `"use server"` files are allowed to be async — they are RPC endpoints, not renderable components.
- The compiler does not check arrow functions assigned to `const`. `<const Posts = async () => ... />` is technically a different shape, but the lint will warn on it.

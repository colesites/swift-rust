# SR0124 — PostCSS configuration exports a function

The PostCSS configuration in your project exports a function instead of a plain object. Swift-Rust's CSS pipeline reads the config as a static shape and ignores function exports, so your plugins silently do not load. The CLI prints this warning to make the situation visible.

## Why this error occurred

PostCSS itself supports both function and object configs. The function form receives a context object (`{ env, file, options }`) and returns the config. The object form is the config directly.

Swift-Rust compiles PostCSS through the Rust `lightningcss` + a JS-side loader. The loader only invokes the object form because it has to resolve the config in a worker thread that does not have access to the same context PostCSS would pass. A function-form config never runs and the result is no PostCSS plugins at all.

The fix is to read the environment from `process.env` and return a static object. This is the same pattern every other JS bundler recommends.

## Reproducing the error

```js
// postcss.config.js
module.exports = ({ env }) => ({
  plugins: {
    "postcss-plugin": env === "production" ? {} : false,
  },
});
```

## Error output

```
[WARN] [SR0124] postcss config exports a function
  --> postcss.config.js:1:18
    |
  1 | module.exports = ({ env }) => ({
    |                  ^^^^^^^^^^^^^
    | Swift-Rust does not invoke function-form PostCSS configs. Your plugins will not run.
    |
    help: return a plain object and read the environment from `process.env`.
```

## How to fix it

1. Convert the function to a plain object:

```js
// postcss.config.js
module.exports = {
  plugins: {
    "postcss-plugin": process.env.NODE_ENV === "production" ? {} : false,
  },
};
```

2. Restart the dev server if it is running.

3. Verify the plugin output by inspecting a built CSS file or running `bun swift-rust info` and looking at the postcss section.

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=css=debug bun swift-rust dev
```

The `css=debug` log prints the resolved PostCSS plugin list and the source of every entry (your config vs. the Swift-Rust built-in defaults).

## Common pitfalls

- The function-form config was popular in old PostCSS tutorials. Newer versions of PostCSS prefer the object form.
- The context argument is sometimes used to read `file` and switch behavior per source file. In Swift-Rust, use a per-file plugin (e.g. a Tailwind variant) instead.
- An ESM `postcss.config.mjs` that does `export default (ctx) => ({...})` triggers the same warning.
- If you actually need per-file context, write the config as a static object and let the plugin read the file path from the standard `postcss` plugin API.

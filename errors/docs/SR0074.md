# SR0074 — Invalid project directory casing

Swift-Rust resolves many paths relative to the project root. On case-sensitive filesystems (Linux, WSL with a Linux distro, case-sensitive APFS volumes) the casing of the path matters: `/Users/me/Project` and `/Users/me/project` point to different directories. On case-insensitive filesystems (default Windows NTFS, default macOS HFS+/APFS) the two are the same — but if you later build the project on a case-sensitive volume, the resolution breaks and the framework refuses to start with this error.

## Why this error occurred

The framework reads the current working directory (`process.cwd()`) and matches it against the resolved real path of the project root. If the two strings differ in case, it raises this error to surface the inconsistency before any file is read.

Common triggers:

- A developer opens PowerShell on Windows, runs `cd path\to\myproject` (lowercase), and starts the dev server. The disk has `MyProject`; the shell has `myproject`.
- A symlink is created with a different case than the target.
- A CI script uses `cd $REPO_DIR` where `$REPO_DIR` is normalized to lowercase by an earlier step.
- A Docker image uses a case-sensitive Linux filesystem but the build script was written on macOS.

The error fires at `bun swift-rust dev` startup. It does not fire on the build command, because the build reads paths from the project config rather than `cwd`.

## Reproducing the error

```bash
# On a case-insensitive filesystem, navigate with a different case
cd /Users/me/myproject  # disk has "MyProject"
bun swift-rust dev
```

## Error output

```
error: [SR0074] invalid project directory casing
  --> current working directory
   |
   = cwd is `/Users/me/myproject` but the real path is
     `/Users/me/MyProject`. case mismatch can break file resolution
     on case-sensitive filesystems.
   = hint: change into the directory using its on-disk casing.
```

## How to fix it

1. Use the exact on-disk case when changing directories.

   ```bash
   cd /Users/me/MyProject
   bun swift-rust dev
   ```

2. On Windows, use PowerShell (`pwsh`) instead of `cmd` — PowerShell preserves case.

3. On macOS, enable case-sensitive APFS only if you are sure the rest of your toolchain handles it. Default APFS is case-insensitive, which is what most apps expect.

4. In CI, normalize the working directory at the start of the job.

   ```bash
   cd "$(realpath .)"
   ```

## Diagnosis

```bash
bun swift-rust doctor
pwd
realpath .
diff <(pwd) <(realpath .)
```

## Common pitfalls

- This error is most often reported on macOS. The default APFS volume is case-insensitive, but a case-sensitive subvolume or an external drive formatted as `Case-sensitive APFS` will catch the mismatch.
- Docker on macOS uses a Linux VM with a case-sensitive ext4 filesystem. Code that worked locally breaks in CI for the same reason.
- The framework does not auto-correct the casing, because doing so could resolve a different directory than the developer intended.
- This is the Swift-Rust analogue of Next.js's `invalid-project-dir-casing`. The fix is identical.

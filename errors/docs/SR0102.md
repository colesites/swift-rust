# SR0102 — NextResponse.next() in a route handler

A route handler in `app/api/.../route.ts` called `NextResponse.next()`. In a route handler, this method has no effect — the handler is the end of the chain, not a pass-through. The framework warns so the call can be replaced with a real response.

## Why this error occurred

`NextResponse.next()` exists for `proxy.ts` (Swift-Rust's equivalent of Next.js middleware): a function that runs before the route handler and can either short-circuit the request or pass it along. The "next" in the name refers to "continue to the next thing in the chain," which is meaningful when there is a next thing.

Inside a route handler, the handler is the terminal step. There is no further thing to forward to — the request was already routed to this handler, and whatever the handler returns is what the client receives. Calling `NextResponse.next()` is a no-op that almost always indicates leftover middleware code that was incorrectly placed in a route handler.

The framework raises a warning rather than a hard error so existing code is not broken, but the warning will become an error in a future major version.

## Reproducing the error

```ts filename="app/api/ping/route.ts"
import { NextResponse } from "swift-rust/server"

export function GET() {
  return NextResponse.next()
}
```

## Error output

```
warn: [SR0102] NextResponse.next() used in a route handler
  --> app/api/ping/route.ts:4:16
   |
 4 |   return NextResponse.next()
   |                ^^^^^ route handlers are the end of the chain; return a Response instead
```

## How to fix it

1. Return a concrete response: `Response.json(...)`, `NextResponse.json(...)`, or a manually constructed `new Response(...)`.
2. If you actually need to intercept and forward, move the logic into `proxy.ts` at the project root.

```ts filename="app/api/ping/route.ts"
import { NextResponse } from "swift-rust/server"

export function GET() {
  return NextResponse.json({ ok: true, ts: Date.now() })
}
```

```ts filename="proxy.ts"
import { NextResponse, type NextRequest } from "swift-rust/server"

export function proxy(req: NextRequest) {
  if (req.headers.get("x-api-key") !== process.env.API_KEY) {
    return new NextResponse("unauthorized", { status: 401 })
  }
  return NextResponse.next()
}
```

## Diagnosis

```bash
bun swift-rust doctor --check route-handlers
RUST_LOG=swift_rust=router=trace bun swift-rust dev
```

`--check route-handlers` runs a static analysis pass on every `route.ts`/`route.tsx` file.

## Common pitfalls

- The `NextResponse.next()` call does not throw at runtime in current versions, so you may not notice the bug in development. The warning is the only signal.
- Some tutorials show `NextResponse.next()` inside a `withAuth` wrapper that the author thought was a route handler. Wrappers in `proxy.ts` are correct; the same wrapper inside a route handler is wrong.
- A route handler that returns `null` (or no return) produces a different error: the framework returns 500 with `SR0005`. The warning for `NextResponse.next()` is the gentler cousin of that case.

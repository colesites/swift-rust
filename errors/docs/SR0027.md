# SR0027 — Invalid resolve alias in build config

A custom bundler extension in `swift-rust.config.json` set `config.resolve.alias` to a brand-new object and dropped the framework's internal aliases. The build aborts because at least one internal module (e.g. the virtual pages module) can no longer be resolved.

## Why this error occurred

Swift-Rust installs a small set of private aliases that the bundler relies on (a virtual entry for the page manifest, the cache runtime, the RSC serializer). When a user's `build.extend` callback replaces the alias map wholesale, those private keys disappear and the next resolve fails with a confusing "module not found" deep in the build.

The framework defends against this by checking that the user-provided alias map still contains the required keys. If any are missing, the build aborts with this code so the fix is obvious.

## Reproducing the error

```json
{
  "name": "site",
  "build": {
    "extend": "config.resolve.alias = { foo: '/abs/path/to/foo' }"
  }
}
```

## Error output

```
[ERROR] [SR0027] invalid resolve alias: required framework aliases are missing
   --> swift-rust.config.json:3:16
   |
 3 |   "extend": "config.resolve.alias = { foo: '/abs/path/to/foo' }"
   |                ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ replace the alias map instead of extending it
   |
  detail: missing required key: '@swift-rust/pages-manifest'
  help: spread the existing alias map: `config.resolve.alias = { ...config.resolve.alias, foo: '/abs/path' }`.
```

## How to fix it

1. Update the `build.extend` callback to spread the existing alias map rather than replace it.
2. Verify the user-defined aliases do not collide with framework-controlled keys.
3. Re-run the build and confirm the error is gone.

```json
{
  "name": "site",
  "build": {
    "extend": "config.resolve.alias = { ...config.resolve.alias, foo: '/abs/path/to/foo' }"
  }
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The trace log prints the full set of required keys and which ones were missing, so you can confirm you have not accidentally removed a key your project actually depends on.

## Common pitfalls

- A spread at the top level (`{ ...config.resolve.alias }`) is not enough if the callback later reassigns a sub-key. Make sure no sub-path is mutated.
- Some third-party bundler presets replace the alias map as part of their setup. Re-run `create-swift-rust` against the current preset to refresh it.
- A `config.resolve` reset (e.g. `config.resolve = {}`) triggers the same error because the entire `alias` key gets dropped.
- The error mentions `@swift-rust/pages-manifest` as an example; the actual missing key may be different in your build.

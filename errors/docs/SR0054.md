# SR0054 — Failed to load `BUILD_ID` from the server

Every Swift-Rust build writes a `BUILD_ID` string into a small static file at `/_swift-rust/static/BUILD_ID`. The client fetches this file once on boot so it can detect when a new version is live. If the file 404s, the client cannot tell whether the current bundle is stale and prints this warning.

## Why this error occurred

The `BUILD_ID` is the link between a client bundle and the server that produced it. When the user clicks a `<Link>` and the client router needs to decide whether the destination page exists in the current bundle, it compares the bundle's embedded `BUILD_ID` against the one served by the live server. A mismatch triggers a soft reload to pick up the new assets.

The warning appears when that round-trip cannot complete:

- The deploy artifact is missing the `BUILD_ID` file because the build was published before it was finalized, or a CDN cache is serving a half-uploaded bundle.
- The origin is returning 5xx for the `BUILD_ID` path because the server is overloaded or the static file route is misconfigured.
- A reverse proxy in front of the app strips the `/_swift-rust/*` prefix.
- The deployment was rolled back to a previous build, and the CDN still serves the new bundle to some users.

This is the Swift-Rust analogue of Next.js's `head-build-id`. The fix path is the same: ensure the `BUILD_ID` file ships with every deploy and is reachable.

## Reproducing the error

```bash
bun swift-rust build
bun swift-rust start
```

```bash
curl -i http://localhost:3000/_swift-rust/static/BUILD_ID
```

## Error output

```
warning: [SR0054] failed to load BUILD_ID from server: 404
  --> GET /_swift-rust/static/BUILD_ID
   |
   = the client cannot determine if a new version is live.
   = the app will keep using the currently loaded bundle, which may be
     out of date. refresh manually if you notice stale UI.
```

## How to fix it

1. Confirm the file is present in the build output.

   ```bash
   ls .swift-rust/build/_swift-rust/static/BUILD_ID
   ```

2. Make sure your reverse proxy or CDN forwards `/_swift-rust/*` to the origin. The required rule is the same one used for any other static asset path.

   ```nginx
   location /_swift-rust/ {
     proxy_pass http://app_origin;
     proxy_set_header Host $host;
   }
   ```

3. If you deploy through the first-party Swift-Rust platform, the platform emits the `BUILD_ID` automatically. If you self-host, your CI must upload the file as part of the artifact.

4. If you are mid-rollout, enable skew protection so that requests from a client built against the previous deploy keep working.

   ```json
   {
     "platform": {
       "skewProtection": true
     }
   }
   ```

## Diagnosis

```bash
bun swift-rust doctor
curl -i http://localhost:3000/_swift-rust/static/BUILD_ID
RUST_LOG=swift_rust=runtime=trace bun swift-rust start
```

## Common pitfalls

- The `BUILD_ID` is fetched only on the first page load, not on every navigation. If you see the warning only after a hard refresh, your CDN is probably stripping it.
- Skew protection is enabled by default on the first-party platform; on a self-hosted setup you have to provide your own grace period by serving both builds behind a sticky cookie.
- This warning is non-fatal. The app keeps running. Do not treat it as a build error; treat it as a deploy hygiene problem.
- The original Next.js error code referenced the Vercel builder. In Swift-Rust the resolution is platform-agnostic.

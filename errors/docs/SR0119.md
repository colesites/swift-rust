# SR0119 — CSP nonce contains invalid characters

A request reached the Swift-Rust server with a Content-Security-Policy header whose `script-src` directive included a `nonce-...` value that contains `<`, `>`, or `&`. Those characters are reserved in the CSP grammar and would let an attacker inject extra directives, so the server refuses to use the nonce for the response.

## Why this error occurred

The CSP specification restricts nonce values to base64-safe characters. The valid alphabet is `A-Z a-z 0-9 + / =` (with `-` and `_` as URL-safe variants). The Swift-Rust server parses every incoming CSP, validates the nonce, and bails out with this error when it sees an illegal character.

The most common root cause is a middlebox (CDN, WAF, edge worker) that HTML-escapes header values as part of its logging or caching layer. The escape converts `<` to `&lt;` in transit, and the downstream server sees a nonce that contains `&`.

## Reproducing the error

```tsx
import { proxy as middleware } from "swift-rust/server";

export function proxy(request: Request) {
  const bad = `nonce-<script />`;
  const csp = `default-src 'self'; script-src 'self' '${bad}'`;
  const response = new Response(null);
  response.headers.set("content-security-policy", csp);
  return response;
}
```

## How to fix it

1. Generate the nonce with a cryptographically random source:

```tsx
import { proxy as middleware } from "swift-rust/server";

export function proxy(request: Request) {
  const nonce = crypto.randomUUID().replace(/-/g, "");
  const csp = `default-src 'self'; script-src 'self' 'nonce-${nonce}'`;
  const response = new Response(null);
  response.headers.set("content-security-policy", csp);
  response.headers.set("x-csp-nonce", nonce);
  return response;
}
```

2. Pass the nonce down to the page so inline scripts can opt in:

```tsx
import { headers } from "swift-rust/headers";

export default async function Page() {
  const nonce = (await headers()).get("x-csp-nonce") ?? "";
  return <script nonce={nonce}>{`console.log("boot")`}</script>;
}
```

3. Verify the upstream proxy is not HTML-escaping the `content-security-policy` header.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=trace bun swift-rust dev
```

The `trace` log prints the parsed CSP and the offending character, including its byte offset in the header value.

## Common pitfalls

- A nonce generated with `Math.random().toString(36)` is too short to be a strong CSP nonce; use `crypto.randomUUID()` or 16 random bytes.
- Base64-encoding HTML markup (for example, `btoa("<script>")`) produces characters such as `+` and `/` that are legal in base64 but the surrounding markup is still wrong — generate from a random byte string instead.
- Some CDNs split long headers across multiple `content-security-policy` lines. The server merges them, but a nonce split across lines is still rejected.
- Setting `unsafe-inline` together with a nonce defeats the purpose. Use one or the other, not both.

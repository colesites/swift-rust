# SR0016 — Babel and swift-rust/font conflict

The project has a custom Babel configuration, which opts the whole project out of the Swift-Rust SWC compiler. The font component (`swift-rust/font`) requires the SWC compiler to bundle and self-host fonts, so the build refuses to proceed.

## Why this error occurred

Swift-Rust's font component generates code at build time that the SWC compiler then transforms and tree-shakes. Babel is a separate transpiler chain; running both on the same source produces inconsistent results and the font loader relies on SWC-specific output (e.g. stable hash renaming, ESM-only emission). To keep the contract simple, the build errors out instead of falling back silently.

## Reproducing the error

```json
{
  "presets": ["next/babel"],
  "plugins": ["@emotion/babel-plugin"]
}
```

```json
{
  "name": "site",
  "compiler": { "swc": true }
}
```

## Error output

```
[ERROR] [SR0016] babel and swift-rust/font conflict: a custom Babel config disables the SWC compiler
   --> .babelrc:1:1
   |
 1 | { "presets": ["next/babel"] }
   | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ remove this file or convert plugins to SWC
   |
  help: delete the Babel config and let SWC compile the project. swift-rust/font requires the SWC pipeline.
```

## How to fix it

1. Delete `.babelrc` (or `babel.config.json`).
2. If a Babel plugin was load-bearing, find the SWC equivalent and add it to `experimental.swcPlugins` in `swift-rust.config.json`.
3. Reinstall the font loader. The component will now bundle correctly.

```json
{
  "name": "site",
  "compiler": {
    "swc": {
      "experimental": {
        "plugins": ["@swc/plugin-emotion"]
      }
    }
  }
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The trace log records which compiler was selected and which file triggered the opt-out, which is helpful when a Babel config is being copied in by a monorepo tool without your knowledge.

## Common pitfalls

- Some CSS-in-JS libraries ship a Babel plugin by default. Check whether the equivalent SWC plugin is available before deleting the Babel config.
- A `.babelrc` deep in `node_modules` does not opt you out — only a config at the project root (or one referenced from a `package.json` in the root) is honored.
- Babel is still useful for non-bundled files (e.g. test runners). Only the **project-level** config is the problem.
- This code is paired with `SR0020` for legacy `@next/font` usage; both can fire on the same project.

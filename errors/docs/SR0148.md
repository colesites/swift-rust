# SR0148 — Threw undefined or null

A `throw` statement in your code produced `undefined` or `null`. JavaScript allows throwing any value, but a non-`Error` throw has no stack trace, no message, and no useful information for debugging. The framework prints this error to flag the pattern.

## Why this error occurred

The runtime can wrap an `Error` in a useful response (with the file, the line, the message). It cannot do the same for `undefined` or `null` — there is no information to display. The framework substitutes a generic message ("threw `undefined`") and prints the location of the `throw` if it can recover it.

The pattern is almost always a bug. A common source is a function that returns `Error | undefined` and a caller that does `throw getError()` without checking which case it received.

## Reproducing the error

```ts
function getData() {
  let error: Error | undefined;
  throw error;
}
```

```tsx
function Page() {
  const error = (data as { error?: Error } | null)?.error ?? null;
  throw error;
}
```

## Error output

```
[ERROR] [SR148] threw `null`
  --> app/page.tsx:5:10
    |
  5 |   throw error;
    |        ^^^^^
    | throwing `null` produces no useful stack trace. throw an `Error` instance instead.
    |
    help: use `throw new Error("description")` and ensure the value is non-null before throwing.
```

## How to fix it

1. Always throw an `Error` instance with a meaningful message:

```ts
function getData() {
  if (!isReady) {
    throw new Error("getData called before initialization");
  }
  return ready();
}
```

2. If a helper may return `undefined`, check the result before throwing:

```ts
const err = getError();
if (err) throw err;
```

3. Audit your codebase for `throw <variable>` patterns and ensure every variable is guaranteed to be an `Error`.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=render=trace bun swift-rust dev
```

`render=trace` prints the throw site and the thrown value (when it can be determined). The doctor output flags any helper that returns a non-`Error` value.

## Common pitfalls

- `throw undefined` and `throw null` look identical at the call site. Always use a named variable or a known `Error` to make the intent clear.
- A caught `Error` that is later re-thrown is fine. The bug is in helpers that *might* return `undefined`.
- The framework prints a "threw `undefined`" or "threw `null`" message; the rest of the trace is the location of the `throw`.
- Some linters warn about `throw`ing non-`Error` values. Enable that rule to catch this pattern at write time.

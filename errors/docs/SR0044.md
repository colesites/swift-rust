# SR0044 — Custom routes not allowed in static export

The project is configured for a static export (`output: 'export'` or `bun swift-rust build --static`) but `swift-rust.config.json` also defines `rewrites`, `redirects`, or `headers`. Static exports do not run a server, so the custom route rules have nothing to attach to. The build warns and ignores them.

## Why this error occurred

A static export emits a fully prerendered bundle that any HTTP server (or CDN) can serve. There is no Swift-Rust process running, so there is no place to evaluate `rewrites`/`redirects`/`headers`. Silently dropping the rules would mean a deploy that looked fine but lacked the routes the project depended on, so the build prints a warning so the regression is visible.

The warning is non-fatal. The export proceeds with the custom rules dropped; you can still ship the result, but the routes will not behave as configured.

## Reproducing the error

```json
{
  "name": "site",
  "output": "export",
  "redirects": [
    { "source": "/old", "destination": "/new", "permanent": true }
  ]
}
```

## Error output

```
[WARN] [SR0044] custom routes ignored in static export: redirects
   --> swift-rust.config.json:3:3
   |
 3 |   "redirects": [...]
   |   ^^^^^^^^^^^^^ custom routes are not applied when `output: 'export'`
   |
  help: remove the `output: 'export'` setting or move the custom routes to a reverse proxy.
```

## How to fix it

1. Decide whether this project is actually static. If it has a server, drop `output: 'export'` and rebuild.
2. If you really need a static export, remove `rewrites`/`redirects`/`headers` from the config.
3. If you need the routing rules anyway, put them on a reverse proxy in front of the static export (Caddy, nginx, Cloudflare, etc.).

```json
{
  "name": "site"
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build --static
```

The trace log lists every custom route rule it skipped during the export, so you can confirm none of them were silently dropped.

## Common pitfalls

- The warning is per rule type, but the build still succeeds. A green build does not mean the rules shipped.
- A `redirects` rule is not the same as a `<Link>` rewrite in code. The former is server-side; the latter is compile-time.
- Route handlers under `app/api/` (SR0012) are a separate concern from custom route rules (SR0044). A static export can have neither.
- A reverse proxy adds latency and a new failure mode. If the rules are critical, prefer a server build.

# SR0135 — Path passed to render() does not start with /

A custom server invoked the internal `render` function with a `path` argument that does not begin with `/`. The router requires a leading slash to disambiguate the path from a route name, so the call is rejected.

## Why this error occurred

Swift-Rust's router treats `/foo` as a path and `foo` as something else entirely (a route alias, in some legacy cases). The internal `render` function takes a `path` parameter; if the value is missing the leading slash, the router cannot resolve it to a route. The framework prints this error and aborts the render.

The check is strict and fires only for the internal `render` API, not for the public routing layer. Custom servers are the only way to trigger it.

## Reproducing the error

```ts
// custom-server.ts
import { createServer } from "swift-rust/server";
import { app } from "./app";

createServer((req, res) => {
  const url = new URL(req.url ?? "/", "http://localhost");
  const path = url.pathname.slice(1);
  app.render(req, res, path);
});
```

When the user requests `/about`, the custom server strips the leading `/` and calls `app.render(req, res, "about")`. The router rejects it.

## Error output

```
[ERROR] [SR0135] render path must start with /
  --> custom-server.ts:6:3
    |
  6 |   app.render(req, res, path);
    |                   ^^^^
    | path: "about"
    | expected: "/about"
    |
    help: prefix the path with `/` before calling `render`.
```

## How to fix it

1. Keep the leading slash:

```ts
import { createServer } from "swift-rust/server";
import { app } from "./app";

createServer((req, res) => {
  const url = new URL(req.url ?? "/", "http://localhost");
  app.render(req, res, url.pathname);
});
```

2. If you need to strip and re-add the slash for any reason, do it explicitly:

```ts
const path = url.pathname.startsWith("/") ? url.pathname : `/${url.pathname}`;
app.render(req, res, path);
```

3. Restart the custom server.

## Diagnosis

```bash
RUST_LOG=swift_rust=router=trace bun swift-rust dev
```

The router trace logs every `render` call with the path, the resolved route, and the eventual response status.

## Common pitfalls

- The error fires only on the internal `render` API. The public routing layer always passes well-formed paths, so the bug is almost always in custom server code.
- `url.pathname` already includes the leading slash. Do not call `.slice(1)` on it.
- Some older templates use `req.url` directly; `req.url` includes the query string. Pass `url.pathname` instead.
- A trailing slash (`/about/`) is allowed and is treated differently from `/about`. The leading-slash check is the only validation.

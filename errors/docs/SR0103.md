# SR0103 — Assigning to `module`

The build refused to compile a chunk that assigned a value to `module`. In a browser or WASM bundle, `module` is a reserved identifier used by the module runtime. Reassigning it would break the runtime's ability to find the chunk's exports.

## Why this error occurred

The bundler produces ESM by default. The output of each chunk uses `module` as the binding for the current module's exports object. If your source code (or an unprocessed CommonJS dependency) writes `module = ...` or `module.exports = ...`, the resulting bundle would clobber the runtime's internal state and every subsequent import from that chunk would fail with a confusing undefined error.

The check is mechanical: the bundler scans the parsed source for any top-level assignment to `module` and rejects the file. This catches both deliberate assignments in app code and accidental inclusions of CommonJS that were not pre-bundled.

The error is reported with the file path and the offending line, but the cause is often a transitive dependency rather than your own code.

## Reproducing the error

```ts filename="lib/legacy.ts"
let module = {
  cache: new Map(),
}

export function memoize<T>(fn: () => T): T {
  if (module.cache.has(fn)) return module.cache.get(fn) as T
  const result = fn()
  module.cache.set(fn, result)
  return result
}
```

A local `module` variable shadows the runtime binding. Even when the intent is innocent, the bundler cannot distinguish your `module` from its own.

## Error output

```
error: [SR0103] no assign module variable: assignment to `module` in lib/legacy.ts
  --> lib/legacy.ts:1:5
   |
 1 | let module = {
   |     ^^^^^^ `module` is a reserved binding in the bundle runtime
```

## How to fix it

1. Rename the variable in your source. `let cacheModule = ...` is usually clearer anyway.
2. If the assignment is in a third-party dependency, pre-bundle the dependency so its CommonJS shim is replaced with an ESM wrapper.
3. If the dependency must run as CommonJS at runtime, add it to the `external` list in `swift-rust.config.json`.

```ts filename="lib/legacy.ts"
const moduleCache = new Map<() => unknown, unknown>()

export function memoize<T>(fn: () => T): T {
  if (moduleCache.has(fn)) return moduleCache.get(fn) as T
  const result = fn()
  moduleCache.set(fn, result)
  return result
}
```

```json filename="swift-rust.config.json"
{
  "bundling": {
    "external": ["some-cjs-only-package"]
  }
}
```

## Diagnosis

```bash
bun swift-rust doctor --check bundle
RUST_LOG=swift_rust=bundling=trace bun swift-rust build
```

`--check bundle` runs the static scan that flags `module` and other reserved identifiers.

## Common pitfalls

- The `module` identifier is fine in `import type` statements; the check fires only on assignments, not on type-only references.
- A `const module = ...` and a `var module = ...` both trip the rule. The bundler does not care about `let` vs `const`.
- A minified dependency that looks like ESM but contains `module.exports` is the classic cause of this error in third-party code. Run `bun swift-rust doctor --check bundle --strict` to see every offending file.
- The bundler's CommonJS interop uses a different identifier internally; renaming your variable to `mod`, `m`, or `mod_` is a quick fix while you investigate.

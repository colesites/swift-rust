# SR0096 — Nested "use cache" needs explicit cacheLife

A function or component marked `"use cache"` wraps another cache with a very short lifetime (`revalidate: 0` or a `cacheLife` shorter than five minutes) without declaring its own `cacheLife`. This is a likely misconfiguration: the outer cache would silently inherit the short lifetime, which is rarely what the author meant.

## Why this error occurred

When a nested `"use cache"` has a very short lifetime, the framework normally treats it as a "dynamic hole" — the inner cache is excluded from the static prerender and is re-fetched per request. That is the intended behavior when the inner cache is the only dynamic piece.

But when the dynamic inner cache lives inside another `"use cache"` without an explicit `cacheLife`, the outer cache's lifetime silently propagates downward: the outer cache becomes just as short-lived as the inner one. That is rarely what was wanted, and it is hard to spot after the fact because the rendered HTML looks correct, only slower.

To prevent this footgun, the compiler requires an explicit `cacheLife()` call on the outer cache whenever it wraps a short-lived inner cache. If you want the outer to stay static, declare a long lifetime. If you want the outer to also be short-lived, declare a short lifetime. Either way, the intent must be in the source.

## Reproducing the error

```tsx filename="components/short-lived-widget.tsx"
import { cacheLife } from "swift-rust/cache"

export async function ShortLivedWidget() {
  "use cache"
  cacheLife("seconds")
  const data = await fetchRealtimeData()
  return <div>{data}</div>
}
```

```tsx filename="app/page.tsx"
import { ShortLivedWidget } from "@/components/short-lived-widget"

export default async function Page() {
  "use cache"
  return (
    <div>
      <h1>Dashboard</h1>
      <p>Last updated: {new Date().toISOString()}</p>
      <ShortLivedWidget />
    </div>
  )
}
```

`Page` is marked `"use cache"` but does not call `cacheLife`. `ShortLivedWidget` does, with a very short lifetime. The compiler refuses to build.

## Error output

```
error: [SR0096] nested "use cache" requires explicit cacheLife on outer
  --> app/page.tsx:5:1
   |
 5 |   "use cache"
   |   ^^^^^^^^^^^ wraps ShortLivedWidget which has cacheLife('seconds'); outer cache must declare its own cacheLife
```

## How to fix it

Add `cacheLife()` to the outer cache. Pick the lifetime that matches your intent.

```tsx filename="app/page.tsx"
import { cacheLife } from "swift-rust/cache"
import { ShortLivedWidget } from "@/components/short-lived-widget"

export default async function Page() {
  "use cache"
  cacheLife("default")
  return (
    <div>
      <h1>Dashboard</h1>
      <p>Last updated: {new Date().toISOString()}</p>
      <ShortLivedWidget />
    </div>
  )
}
```

Or, if the outer should also be short-lived, declare it explicitly and wrap the dynamic piece in `<Suspense>`:

```tsx filename="app/page.tsx"
import { Suspense } from "react"
import { cacheLife } from "swift-rust/cache"
import { ShortLivedWidget } from "@/components/short-lived-widget"

async function Content() {
  "use cache"
  cacheLife("seconds")
  return (
    <>
      <p>Last updated: {new Date().toISOString()}</p>
      <ShortLivedWidget />
    </>
  )
}

export default function Page() {
  return (
    <div>
      <h1>Dashboard</h1>
      <Suspense fallback={<p>Loading...</p>}>
        <Content />
      </Suspense>
    </div>
  )
}
```

## Diagnosis

```bash
bun swift-rust doctor --check caches
RUST_LOG=swift_rust=cache=trace bun swift-rust build
```

`--check caches` walks the dependency graph and reports every place a `"use cache"` lacks an explicit `cacheLife`.

## Common pitfalls

- A short lifetime is anything with `revalidate: 0` or `cacheLife('seconds')` (or `'minutes'` with a sub-5-minute value). Long lifetimes like `'hours'`, `'days'`, `'max'`, and a custom value over five minutes are not affected.
- The check is structural: the outer cache is examined for any descendant that is also cached with a short lifetime. The error fires even if the inner cache would never actually run.
- `"use cache: remote"` is a separate directive used for serverless runtimes that don't persist in-memory state across requests. The rule still applies.

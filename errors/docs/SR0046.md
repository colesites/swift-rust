# SR0046 — SWC failed to load

Swift-Rust compiles TypeScript and TSX through SWC, a Rust-based compiler. SWC ships as a native binary that the package manager must download for your specific platform. If that download fails, the binary is for the wrong architecture, or it was stripped from the lockfile, the compiler cannot start and the build aborts with this error.

## Why this error occurred

SWC replaces the slower Babel toolchain that older Next.js installations relied on. Unlike Babel, SWC is not pure JavaScript — it depends on a native `.node` artifact (or platform-equivalent shared library) produced by `napi-rs`. The artifact is published as an `optionalDependency` keyed on a triple such as `darwin-arm64`, `linux-x64-gnu`, or `win32-x64-msvc`.

This error appears when the Swift-Rust toolchain asks Node/Bun to load `@swc/core` and the `dlopen` call fails. Common root causes are:

- An outdated lockfile (e.g. generated on `darwin-x64`) does not contain the `darwin-arm64` variant, so when you switch to an M-series Mac the binary is missing.
- The package manager was invoked with `--no-optional`, which silently skips the platform binary.
- A corporate proxy or sandbox blocked the download of the binary tarball.
- On Windows, the Microsoft Visual C++ Redistributable is not installed, so the binary cannot resolve its runtime symbols.
- An ARM-vs-x86 mismatch between the active Node version and the installed binary.

## Reproducing the error

```bash
bun install --no-optional
bun swift-rust build
```

## Error output

```
error: [SR0046] SWC failed to load: could not dlopen @swc/core (darwin-arm64)
  --> node_modules/@swc/core/index.js:1:1
   |
 1 | throw new Error("Cannot find module '@swc/core-darwin-arm64'");
   | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  |
  = hint: ensure the platform-specific optionalDependency was installed.
          run `bun install --force` to refresh the lockfile.
```

## How to fix it

1. Refresh the dependency graph so the platform binary is reinstalled.

   ```bash
   rm -rf node_modules bun.lockb
   bun install
   ```

2. On Windows, install the [Microsoft Visual C++ Redistributable](https://docs.microsoft.com/en-us/cpp/windows/latest-supported-vc-redist).

3. If your network blocks the CDN, allowlist `https://registry.npmjs.org` and `https://binaries.swc.rs` in your proxy.

4. As a last resort, opt out of SWC by enabling the Babel fallback in `swift-rust.config.json`.

   ```json
   {
     "compiler": {
       "useBabel": true
     }
   }
   ```

   And add a `.babelrc` at the project root:

   ```json
   {
     "presets": ["swift-rust/babel"]
   }
   ```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=trace bun swift-rust build
ls node_modules/@swc/core-* 2>/dev/null || echo "no platform binary installed"
node -p "process.platform + '-' + process.arch"
```

## Common pitfalls

- Switching from an x64 Node version to an arm64 one (or vice versa) on Apple Silicon requires reinstalling dependencies — Bun will not re-evaluate `optionalDependencies` automatically.
- Docker images built on an x64 host do not ship the linux-arm64 binary; build multi-arch images with `docker buildx` and a per-arch base.
- This error used to surface as `Cannot find module '@next/swc-linux-x64-gnu'` in Next.js. The Swift-Rust variant uses `@swc/core` as the binary package directly.
- The `bun install --no-optional` flag is the most common cause in CI pipelines that copy flags from npm scripts.

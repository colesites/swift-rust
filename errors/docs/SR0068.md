# SR0068 — Invalid `images` config

The `image` block in `swift-rust.config.json` configures the `<Image>` component from `swift-rust/image`. It controls which device sizes and image sizes are emitted, which remote hosts are allowed, what format the response is encoded in, and how the cache is sized. The schema is strict: a value of the wrong type, an unknown key, or a value that exceeds a documented limit aborts the build.

## Why this error occurred

The image pipeline is one of the most performance-sensitive parts of the framework. To keep the build deterministic and the optimizer safe, the schema is enforced at config-load time rather than at first use. Common causes:

- A typo in a key name (`device_size` instead of `deviceSizes`).
- An array that exceeds its maximum length (25 `deviceSizes`, 25 `imageSizes`, 50 `domains`, 50 `remotePatterns`, 20 `qualities`).
- A `loader` set to a value other than `'default' | 'imgix' | 'cloudinary' | 'akamai' | 'custom'`.
- A `contentSecurityPolicy` that is not a string.
- A `minimumCacheTTL` that is negative or not an integer.

The full list of accepted fields and their limits is below.

## Reproducing the error

```json
{
  "image": {
    "deviceSizes": [640, 750, 828, 1080, 1200, 1920, 2048, 3840],
    "imageSizes": [16, 32, 48, 64, 96, 128, 256, 384],
    "loader": "imgproxy"
  }
}
```

## Error output

```
error: [SR0068] invalid images config
  --> swift-rust.config.json:5:3
   |
 5 |     "loader": "imgproxy"
   |              ^^^^^^^^^^^
   |
   = unknown loader "imgproxy". expected one of: default, imgix,
     cloudinary, akamai, custom. if you need a custom loader, set
     `loaderFile` instead.
```

## How to fix it

1. Use a known loader name, or a `loaderFile` for custom loaders.

   ```json
   {
     "image": {
       "deviceSizes": [640, 750, 828, 1080, 1200, 1920, 2048, 3840],
       "imageSizes": [32, 48, 64, 96, 128, 256, 384],
       "path": "/_swift-rust/image",
       "loader": "default",
       "loaderFile": "",
       "minimumCacheTTL": 14400,
       "formats": ["image/webp"],
       "dangerouslyAllowSVG": false,
       "contentSecurityPolicy": "default-src 'self'; script-src 'none'; sandbox;",
       "contentDispositionType": "attachment",
       "localPatterns": [],
       "remotePatterns": [],
       "qualities": [25, 50, 75],
       "unoptimized": false
     }
   }
   ```

2. Trim any arrays that exceed their limits. The defaults shown above are the recommended starting point.

3. If you previously used the `domains` field, migrate to `remotePatterns`. `domains` is deprecated and prints a warning.

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info | grep -A 20 '"image"'
RUST_LOG=swift_rust=image=trace bun swift-rust build
```

## Common pitfalls

- The `loader` value `'custom'` requires `loaderFile` to point at a module exporting a default function with the signature `({ src, width, quality }) => string`.
- `dangerouslyAllowSVG: true` is allowed but emits a build-time warning; only enable it if you trust the SVG sources.
- The schema is camelCase in JSON, kebab-case in TOML, and snake_case in YAML — the framework normalizes these.
- The error fires at config load, not at first image request. The first request after a bad config never happens because the dev server refuses to start.

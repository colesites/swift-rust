# SR0141 — Sharp is not installed

The image optimization pipeline tried to load `sharp` (the native image-processing library) and the package could not be resolved. The pipeline falls back to the slower JavaScript implementation during dev, but refuses to run in production with a degraded image optimizer, so it errors out at startup.

## Why this error occurred

`<Image>` from `swift-rust/image` uses `sharp` to resize, re-encode, and serve optimized images in production. `sharp` is a native addon (a Rust crate with N-API bindings) that compiles at install time. The framework requires it as a regular dependency for any production-mode start.

Three common reasons it is missing:

- The package was not added to `package.json`. Dev-only `sharp` installations are stripped in many deployment targets.
- The lockfile was generated without `sharp` and `bun install` skipped it (some `--production` and `--frozen-lockfile` workflows do this).
- A custom Docker layer or build step excludes `sharp` from the final image.

The framework prints this error at server start, names the resolution path it tried, and refuses to start the image pipeline.

## Reproducing the error

```bash
bun swift-rust start
```

```log
[ERROR] [SR0141] sharp is not installed
  --> .
    |
    | image optimization requires `sharp` in production mode.
    | looked for: ./node_modules/sharp
    |
    help: run `bun add sharp` and rebuild.
```

## Error output

```
[ERROR] [SR0141] sharp is not installed
  --> .
    |
    | image optimization requires `sharp` in production mode.
    | looked for: ./node_modules/sharp
    |
    help: run `bun add sharp` and rebuild.
```

## How to fix it

1. Install `sharp` as a regular dependency:

```bash
bun add sharp
```

2. If `sharp` is installed but not in the default resolution path (for example, you keep it in a vendored directory), point Swift-Rust at it:

```bash
export SHARP_PATH=/path/to/node_modules/sharp
bun swift-rust start
```

3. Re-run the production build and restart the server:

```bash
bun swift-rust build
bun swift-rust start
```

## Diagnosis

```bash
bun swift-rust doctor
bun pm ls sharp
```

`bun pm ls sharp` shows the resolved path; if it is missing, the doctor output flags it. The doctor also reports whether the native binary inside `sharp` is loadable for the current OS / architecture.

## Common pitfalls

- `sharp` is a native module. It must be reinstalled (not just copied) when you change the OS or CPU architecture. A `node_modules` directory from an x86 build will not load on ARM.
- Some monorepo tools hoist `sharp` to a workspace root. Make sure the package that imports it has access to the hoisted version.
- The `sharp` package pulls in a few platform-specific binaries. If you are using Bun's `--production` flag, Bun will install only the runtime dependencies and skip the platform package. Reinstall with the platform package.
- A Vercel-like platform installs `sharp` automatically, but a self-hosted target has to install it explicitly. The error message is the same.

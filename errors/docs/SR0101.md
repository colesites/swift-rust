# SR0101 — Request read inside "use cache"

A function or component marked `"use cache"` tried to read the incoming request — by calling `cookies()`, `headers()`, or `draftMode()` from `swift-rust/headers`. The build refuses to compile this combination because it would silently invalidate the cache on every request, which is almost never what was intended.

## Why this error occurred

The whole point of `"use cache"` is that the function's return value is keyed by its arguments and reused across requests. A function that reads from the request would have a different return value for every request, which means the cache provides no benefit and the function still pays the cache lookup cost on every call.

The framework cannot tell whether the request read is "just for logging" (in which case it would be safe) or "for deciding what to return" (in which case it would silently break the cache). Rather than guess, the build rejects the pattern outright. You can always opt out by removing the `"use cache"` directive.

## Reproducing the error

```tsx filename="app/page.tsx"
import { cookies } from "swift-rust/headers"

async function getExampleData() {
  "use cache"
  const isLoggedIn = (await cookies()).has("token")
  const posts = await db.post.findMany({ where: isLoggedIn ? { draft: false } : undefined })
  return posts
}

export default async function Page() {
  const data = await getExampleData()
  return <pre>{JSON.stringify(data)}</pre>
}
```

`getExampleData` is cached but reads `cookies()`.

## Error output

```
error: [SR0101] request read inside "use cache": cookies() called from getExampleData
  --> app/page.tsx:5:28
   |
 5 |   const isLoggedIn = (await cookies()).has("token")
   |                      ^^^^^^^^^^^ cookies() cannot be used inside "use cache"
```

## How to fix it

Move the request read outside the cached function and pass the value in as an argument. The argument becomes part of the cache key.

```tsx filename="app/page.tsx"
import { cookies } from "swift-rust/headers"

async function getExampleData(isLoggedIn: boolean) {
  "use cache"
  const posts = await db.post.findMany({ where: isLoggedIn ? { draft: false } : undefined })
  return posts
}

export default async function Page() {
  const isLoggedIn = (await cookies()).has("token")
  const data = await getExampleData(isLoggedIn)
  return <pre>{JSON.stringify(data)}</pre>
}
```

If the function truly must read the request, remove the directive:

```tsx filename="app/page.tsx"
import { cookies } from "swift-rust/headers"

async function getExampleData() {
  const isLoggedIn = (await cookies()).has("token")
  const posts = await db.post.findMany({ where: isLoggedIn ? { draft: false } : undefined })
  return posts
}
```

## Diagnosis

```bash
bun swift-rust doctor --check caches
RUST_LOG=swift_rust=cache=trace bun swift-rust build
```

`--check caches` flags every cached function that calls a request-reading helper.

## Common pitfalls

- Reading `headers().get('x-request-id')` for tracing is also rejected, even though it does not affect the response. Move the read above the cache boundary.
- A `cookies()` call inside a `<Suspense>` fallback is fine — the fallback is not cached.
- `draftMode()` is treated the same as `cookies()`: it must not be read inside a cached function.
- The error is about the function's static shape. A function that imports a helper but never calls it is fine; the build does not flag the import alone.

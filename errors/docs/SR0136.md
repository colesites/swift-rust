# SR0136 — Reserved page prop

A page's data hook returned an object that includes a key reserved by the framework (`url`). Swift-Rust uses that key to inject request-time data, and a user-supplied value would shadow the framework value, so the load fails with this error.

## Why this error occurred

The framework reserves a small set of keys on the data object passed to a page. The most common one is `url`, which the framework uses to expose the current request URL to the page. If user code also returns a `url` field, the framework cannot decide which value to use, so it refuses to start.

The set of reserved keys is internal and may grow; the framework intentionally surfaces the error rather than silently dropping the field, so you know to rename it.

## Reproducing the error

```ts
// app/post/[id]/page.tsx
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const res = await fetch(`https://api.example.com/posts/${id}`);
  const data = (await res.json()) as { url: string; title: string };
  return <PostPage url={data.url} title={data.title} />;
}
```

If `PostPage` accepts a `url` prop, the framework rejects the page because `url` is reserved.

## Error output

```
  --> app/post/[id]/page.tsx:6:24
    |

    |                        ^^^^^^^^^^
    | `url` is reserved by the framework and cannot be passed as a page prop.
    |
    help: rename the prop or use the `usePathname()` / `useSearchParams()` hook to read URL info.
```

## How to fix it

1. Rename the prop in your data and components:

```ts
// app/post/[id]/page.tsx
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const res = await fetch(`https://api.example.com/posts/${id}`);
  const data = (await res.json()) as { url: string; title: string };
  return <PostPage canonicalUrl={data.url} title={data.title} />;
}
```

2. If you need to read the current URL inside a client component, use the router hooks:

```tsx
"use client";

import { usePathname, useSearchParams } from "swift-rust/navigation";

export function CurrentUrl() {
  const path = usePathname();
  const params = useSearchParams();
  return <p>{`${path}?${params.toString()}`}</p>;
}
```

3. In a server component, the request URL is exposed via `headers()`:

```tsx
import { headers } from "swift-rust/headers";

export default async function Page() {
  const h = await headers();
  return <p>{h.get("x-url")}</p>;
}
```

## Diagnosis

```bash
RUST_LOG=swift_rust=router=trace bun swift-rust dev
```

The router trace logs the props object shape for every page render and flags reserved keys.

## Common pitfalls

- The reserved key set includes more than `url`; the error always names the exact key.
- A page can pass a `url` prop to a child component — the check is on the *page's* props, not on arbitrary JSX.
- Renaming the field in the data object is the only fix. The framework does not provide a "disable" switch.
- If the value comes from a third-party API and you cannot rename the field, destructure it into a different variable before passing it to JSX.

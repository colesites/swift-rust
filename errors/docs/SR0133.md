# SR0133 â€” Hydration mismatch

The HTML produced by the server (used as the initial document) does not match the React tree the client produces during the first render. React's hydration step notices the divergence, throws, and asks the client to recover by re-rendering the entire tree from scratch. The error is logged with the exact text and the first differing position.

## Why this error occurred

Hydration is the moment when React takes over the prerendered HTML and attaches event handlers. The first client render must produce a tree whose HTML output is byte-for-byte identical to what the server sent, or React considers the tree "unrecoverable" and gives up.

Common reasons for a divergence:

- The render logic uses `Date()` or `Math.random()`, which produce different values on the server and the client.
- The render logic reads `window` or `localStorage`, which do not exist on the server.
- An iOS-style auto-detection of phone numbers / dates wraps text nodes in `<a>` tags after the server renders but before hydration runs.
- A CDN rewrites the HTML (auto-minify, BOM injection, attribute reordering) and the client no longer matches.
- The component wraps a `<div>` inside a `<p>`, or an `<a>` inside another `<a>`, and the browser silently re-parenthesizes the DOM.
- A browser extension (Grammarly, Dark Reader) injects nodes before hydration.

The error message names the first node that differs, which is usually enough to track the bug to its source.

## Reproducing the error

```tsx
export default function Clock() {
  return <time dateTime="2026-06-02">{new Date().toLocaleString()}</time>;
}
```

The server renders a timestamp, the client renders a different one, and React complains.

```tsx
export default function BrowserCheck() {
  if (typeof window !== "undefined") {
    return <p>browser</p>;
  }
  return <p>server</p>;
}
```

The server renders `server`; the client renders `browser`; React complains.

## How to fix it

1. Move the dynamic read into a `useEffect`:

```tsx
import { useEffect, useState } from "react";

export default function Clock() {
  const [now, setNow] = useState<string | null>(null);
  useEffect(() => {
    setNow(new Date().toLocaleString());
    const id = setInterval(() => setNow(new Date().toLocaleString()), 1000);
    return () => clearInterval(id);
  }, []);
  return <time dateTime="2026-06-02">{now ?? "â€”"}</time>;
}
```

2. For intentional divergences (e.g. timestamps), silence the warning on the specific element:

```tsx
<time dateTime="2026-06-02" suppressHydrationWarning>
  {new Date().toLocaleString()}
</time>
```

3. For components that depend entirely on the browser, mark them client-only and skip SSR:

```tsx
"use client";

import dynamic from "react";

const NoSSR = dynamic(() => import("./browser-only"), { ssr: false });
```

4. Disable CDN HTML auto-minify (Cloudflare, Akamai, etc.) so the document that reaches the client is byte-for-byte identical to the server output.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=hydrate=trace bun swift-rust dev
```

`hydrate=trace` prints the full server-rendered tree and the client-rendered tree, with the first divergent node highlighted.

## Common pitfalls

- `suppressHydrationWarning` only silences the warning for the element it is set on, not for its children. Apply it to the specific element that intentionally differs.
- iOS Safari auto-links phone numbers. Disable with `<meta name="format-detection" content="telephone=no, date=no, email=no, address=no" />` in the root layout.
- `Date()` and `Math.random()` are not the only offenders. `crypto.randomUUID()` and `performance.now()` also differ.
- Browser extensions are hard to detect. The fastest mitigation is to install the affected extension in a clean browser profile and check whether the warning disappears.

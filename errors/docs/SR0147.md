# SR0147 — Synchronous dynamic API access

A dynamic API — `params`, `searchParams`, `cookies()`, `draftMode()`, `headers()` — was accessed synchronously. These APIs return a `Promise` in Swift-Rust (matching the React 19 / Next 15 design) and must be awaited before their values can be read. Synchronous access is a compile-time error.

## Why this error occurred

Dynamic APIs depend on request-time data and cannot be evaluated during static analysis. The framework returns a `Promise` so the call site can opt into dynamic rendering explicitly (by awaiting) or opt into static rendering (by passing the Promise through to a child that does the await). Synchronous access would force dynamic rendering at the call site, which breaks the ability to statically render as much of the tree as possible.

The compile-time check is strict: any direct read (`.id`, `[Symbol.iterator]()`, `for ... of`, `{...spread}`) on the return value is rejected. The migration is mechanical — add `async`, add `await` — and a codemod handles the bulk of it.

## Reproducing the error

```tsx
function Page({ params }: { params: Promise<{ id: string }> }) {
  return <p>ID: {params.id}</p>;
}
```

```tsx
import { cookies } from "swift-rust/headers";

function Banner() {
  const c = cookies();
  return <p>{c.get("name")?.value}</p>;
}
```

## Error output

```
[ERROR] [SR0147] synchronous access of dynamic API
  --> app/posts/[id]/page.tsx:2:10
    |
  2 |   return <p>ID: {params.id}</p>;
    |                  ^^^^^^^^^
    | `params` is a Promise; use `await` to read its value, or `React.use()` in a client component.
    |
    help: declare the function `async` and `await params` before reading.
```

## How to fix it

1. In a server component, add `async` and `await` the API:

```tsx
export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  return <p>ID: {id}</p>;
}
```

2. In a client component, use `React.use()` to unwrap the Promise:

```tsx
"use client";

import * as React from "react";

export default function Page({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = React.use(params);
  return <p>ID: {id}</p>;
}
```

3. Run the codemod to handle the bulk of the migration:

```bash
bunx @swift-rust/codemod async-request-api .
```

The codemod handles the common patterns. Anything it cannot migrate is left with a `@swift-rust-codemod-error` comment that the build will surface until addressed.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=render=trace bun swift-rust dev
```

`render=trace` prints the dynamic API call sites and the resolved values at request time, which makes it easy to confirm the migration.

## Common pitfalls

- Spreading a dynamic API (`{...params}`) is the same as reading every key. The check catches it.
- The codemod cannot infer types; it adds the `await` and lets you fix the type by hand.
- `React.use` is only valid in client components. In a server component, you must `await`.
- Some libraries cache the return value of `cookies()` in a module-scope variable. That variable will be `undefined` until the first request; await the API at the call site instead.

# SR0121 — Page data collection timed out

During a production build, Swift-Rust walks the route table and asks every server component for its data. If the worker pool that drives that walk makes no progress for the configured timeout window, it restarts. After several restarts with the same job still incomplete, the build fails with this error.

## Why this error occurred

The data-collection step is the phase where the bundler resolves all imports for a page, evaluates the server component tree, and stores the static-renderable parts of the output. The workers are isolated so that one slow route cannot stall every other route. When a worker appears stuck, Swift-Rust cycles it.

If a single page repeatedly fails to make progress (its `await` chain never settles, a fetch hangs, or it accidentally enters a busy loop), the worker is restarted but the same job is re-attempted, and the same hang recurs. After a few cycles the build gives up.

The default per-page timeout is 60 seconds. Routes that legitimately need more — for example, an index that pulls a million-row table — have to opt in to a higher value.

## Reproducing the error

```tsx
async function fetchProducts() {
  return new Promise(() => {});
}

export default async function Page() {
  const products = await fetchProducts();
  return <ul>{products.map((p) => <li key={p.id}>{p.name}</li>)}</ul>;
}
```

## Error output

```
[ERROR] [SR0121] page data collection timed out after multiple attempts
  --> app/products/page.tsx:1:1
    |
    | route: /products
    | attempts: 3
    | total wait: 180.0s
    | cause: the server component did not resolve within the configured timeout.
    |
    help: add a fetch timeout, ensure all Promises resolve, or raise `build.staticPageTimeout`.
```

## How to fix it

1. Add a timeout to any external fetch:

```tsx
async function fetchProducts() {
  const res = await fetch("https://api.example.com/products", {
    signal: AbortSignal.timeout(10_000),
  });
  if (!res.ok) throw new Error(`upstream ${res.status}`);
  return res.json();
}

export default async function Page() {
  const products = await fetchProducts();
  return <ul>{products.map((p) => <li key={p.id}>{p.name}</li>)}</ul>;
}
```

2. Make sure the route's data is reachable. A `Promise` that never settles is the most common cause.

3. If the workload is genuinely slow, raise the timeout:

```json
{
  "build": {
    "staticPageTimeout": 300
  }
}
```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=trace bun swift-rust build
```

The `trace` log prints the worker lifecycle (start, idle, restart, give up) with timestamps. Search for `data_collection` to find the slow route.

## Common pitfalls

- A `fetch` without `signal: AbortSignal.timeout(...)` waits for the platform default (often several minutes). Always pass a signal.
- A `setInterval` or `while (true)` inside a server component never yields control. Use bounded iteration.
- The timeout is per attempt, not per build. A route that takes 30 seconds will succeed on the first try even with the default 60-second window.
- Slow data and slow rendering are different. If only rendering is slow, you will see a different code (SR0128).

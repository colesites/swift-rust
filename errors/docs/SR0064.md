# SR0064 — Invalid options type in a dynamic import

Swift-Rust's dynamic component helper accepts a second argument that is the options object. The framework requires the object to be a literal so that it can analyze the call site at build time and decide whether to ship a server-rendered shell, a client-only chunk, or both. A non-literal options value makes that analysis impossible, and the helper refuses to compile.

## Why this error occurred

The dynamic helper does not just forward `options` to `React.lazy` or to a server-render boundary — it inspects keys like `ssr`, `loading`, `suspense`, and a few framework-specific flags to decide how the imported module is bundled. If the options are behind a variable, the bundler cannot know what they are and has to fall back to the most conservative behavior (server-render the import, no loading state, no suspense). The framework instead raises this error to make the limitation visible.

Common causes:

- The options are extracted into a `const options = { ssr: false }` and passed by reference.
- A wrapper function is added that takes a config object and forwards it.
- A linter or formatter auto-converted the literal into a variable.

This is the Swift-Rust analogue of Next.js's `invalid-dynamic-options-type`. The fix is the same: keep the options inline.

## Reproducing the error

```tsx
import { dynamic } from "swift-rust/dynamic";

const options = { ssr: false };
const HeavyChart = dynamic(() => import("./HeavyChart"), options);
```

## Error output

```
error: [SR0064] invalid options type in a dynamic import
  --> app/dashboard/page.tsx:4:1
   |
 4 | const options = { ssr: false };
   | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
   |
   = the second argument to `dynamic` must be an object literal at the
     call site. dynamic analysis of the options is required to decide
     how to bundle the import.
```

## How to fix it

1. Pass the options inline.

   ```tsx
   import { dynamic } from "swift-rust/dynamic";

   const HeavyChart = dynamic(() => import("./HeavyChart"), {
     ssr: false,
   });
   ```

2. If you need a reusable preset, export a function that returns the dynamic call.

   ```tsx
   import { dynamic } from "swift-rust/dynamic";

   export const clientOnly = <P extends {}>(loader: () => Promise<{ default: React.ComponentType<P> }>) =>
     dynamic(loader, { ssr: false });
   ```

   ```tsx
   import { clientOnly } from "../lib/dynamic";

   const HeavyChart = clientOnly(() => import("./HeavyChart"));
   ```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=dynamic=trace bun swift-rust build
```

## Common pitfalls

- The error is only raised when the framework can prove the options are not a literal. Spreading an object literal is fine; spreading a variable is not.
- If you have a long list of options and want to keep them readable, comment them inline in the object literal — the analyzer only cares that the value is statically known.
- This error is the modern equivalent of Next.js's `next/dynamic` validation. The behaviour matches.

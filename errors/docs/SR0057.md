# SR0057 — ESM packages must be imported, not required

Many modern npm packages — including `node-fetch` 3+, `chalk` 5+, `execa` 6+, and most of the new wave of "pure ESM" libraries — are published without a CommonJS entry point. They can only be loaded with the ESM `import` statement (or a dynamic `import()`). The Swift-Rust bundler, running on Bun, refuses to silently rewrite a `require()` call into an `import()` because doing so would change execution order and break top-level await.

## Why this error occurred

The error appears when the bundler traces a `require()` call to a package whose `package.json` declares `"type": "module"` and has no `exports.require` condition. The most common sources of `require` in an otherwise ESM project are:

- A TypeScript or Babel config that compiles `import` down to `require`. This was the default for years; if the project was scaffolded with `module: "commonjs"` in `tsconfig.json` it still happens.
- A custom server runtime file that uses `require` directly. The dev server tolerates this in `pages/`-era projects, but the `app/` server is strictly ESM.
- A CommonJS dependency in your tree that does a `require('esm-pkg')` and is itself loaded by your code.
- Webpack-style `require.ensure` or `require.context` calls left over from a migration.

This is the Swift-Rust analogue of Next.js's `import-esm-externals`. The fix is the same: keep imports as `import` all the way down.

## Reproducing the error

```ts
// app/api/route.ts
import type { RequestHandler } from "swift-rust/server";

const fetch = require("node-fetch");

export const GET: RequestHandler = async () => {
  const res = await fetch("https://example.com");
  return new Response(await res.text());
};
```

## Error output

```
warning: [SR0057] ESM packages must be imported, not required
  --> app/api/route.ts:4:1
   |
 4 | const fetch = require("node-fetch");
   |                ^^^^^^^^^^^^^^^^^^^
   |
   = `node-fetch` is an ESM-only package. `require()` cannot load it.
   = hint: use `import fetch from "node-fetch"` (or `await import("node-fetch")`).
```

## How to fix it

1. Switch to ESM imports.

   ```ts
   import type { RequestHandler } from "swift-rust/server";
   import fetch from "node-fetch";

   export const GET: RequestHandler = async () => {
     const res = await fetch("https://example.com");
     return new Response(await res.text());
   };
   ```

2. Make sure your `tsconfig.json` keeps ESM syntax.

   ```json
   {
     "compilerOptions": {
       "module": "esnext",
       "moduleResolution": "bundler"
     }
   }
   ```

3. If you depend on a CommonJS package that internally does `require('esm-pkg')`, the only escape hatch is to enable loose ESM externals in the project config. This is a workaround, not a fix.

   ```json
   {
     "bundler": {
       "esmExternals": "loose"
     }
   }
   ```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=bundler=trace bun swift-rust build
cat package.json | bunx json-cli '."type"'
```

## Common pitfalls

- The error fires at bundle time, not at runtime. A `require` that is never actually called (e.g. behind a `if (false)` guard) is still flagged.
- `"type": "module"` in a package's `package.json` is the canonical signal. The error will not fire for a package that ships both CJS and ESM entries.
- Dynamic `import('esm-pkg')` is always fine, including inside a CommonJS file. The bundler hoists the call to be a top-level chunk.
- The original Next.js documentation also suggested `experimental.esmExternals: 'loose'`. The Swift-Rust equivalent is `bundler.esmExternals`. Use it as a last resort only.

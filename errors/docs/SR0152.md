## Why this error occurred

The <Pdf> viewer fetches and parses the PDF in the browser using PDF.js. The fetch can fail for reasons that are independent of the PDF itself: a typo in the URL, a CORS preflight rejection, a proxy that serves an HTML error page, or a network outage. The viewer reports the failure through three channels:

1. The on-screen error UI (built-in unless `errorFallback` is provided).
2. The `onError` callback, which receives a `PdfError` with `code: "SR0152"` and `kind: "load" | "network" | "invalid"`.
3. The thrown error during SSR, if the component is forced to render on the server (it normally renders a download link instead — see `fallback`).

## Reproducing the error

```tsx
<Pdf src="/missing.pdf" />
```

The viewer renders an error panel with a `Retry` button and an `Open in new tab` link. The browser console logs:

```
PdfError [SR0152]: Failed to fetch
    at <Pdf> (packages/pdf/src/viewer.tsx:218:21)
```

## Error output

```
PdfError: load
  code: "SR0152"
  kind: "load"
  url: "/missing.pdf"
  cause: TypeError: Failed to fetch
```

The `kind` field discriminates:

| `kind` | Meaning |
|---|---|
| `"load"` | Generic load failure (caught any other reason) |
| `"network"` | Network error (offline, DNS, CORS) |
| `"invalid"` | The bytes were fetched successfully but are not a valid PDF (wrong Content-Type, password-protected, truncated) |

## How to fix it

1. **Verify the URL.** Open it in a new tab. The browser will either download the PDF or render it. If neither happens, the URL is wrong.

2. **Check `Content-Type`.** The server should respond with `application/pdf`. Some misconfigured CDNs serve `application/octet-stream`; PDF.js will still parse it but some browsers will refuse to fetch it cross-origin without a CORS header.

3. **Add CORS headers if cross-origin.** The viewer fetches the PDF as an `ArrayBuffer`; the response must include `Access-Control-Allow-Origin: <your-origin>` (or `*`).

4. **Strip passwords.** If the PDF is encrypted, decrypt it server-side (e.g. with `qpdf --decrypt`) and serve the unencrypted version. PDF.js does not accept a password in the current viewer.

5. **Provide a fallback for offline / known-bad sources.** Pass `fallback={<a href={src}>Download PDF</a>}` to control the SSR and pre-hydration markup.

## Diagnosis

```ts
<Pdf
  src="/file.pdf"
  onError={(err) => {
    if (err.kind === "network") analytics("pdf.network");
    if (err.kind === "invalid") analytics("pdf.invalid");
  }}
/>
```

The `url` field on the error tells you which `src` failed when the same component is reused with different sources.

## Common pitfalls

- **Using a `file://` URL.** PDF.js refuses to fetch `file://` URLs from a web origin. Serve the PDF over HTTP.
- **Using an SPA fallback route.** Some routers return `index.html` with a 200 status for unknown paths. The viewer treats that as a load failure, not a parse failure, because the `Content-Type` is `text/html` but the body is also not a PDF. The fix is to make the static-file server return 404 for missing files.
- **Assuming the error message is the URL.** It is not. Inspect `err.url` for the source and `err.message` for the underlying reason.

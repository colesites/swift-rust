# SR0062 — Invalid API route status/body response

The HTTP specification requires that responses with status `204 No Content` and `304 Not Modified` have no body. Swift-Rust enforces this contract at the server layer: a handler that sets one of these statuses and also returns a body is rejected before the response is sent, and this error is raised in the server log.

## Why this error occurred

The framework is strict about HTTP semantics to make caching and client behaviour predictable. A `204` means "I did the work, there is nothing to send back"; a `304` means "your cached copy is still good, do not re-fetch the body". Sending a body alongside either status is undefined behaviour in the spec — most clients ignore it, some libraries throw, and proxies may strip it.

The most common sources of the mistake:

- A developer calls `Response.json(payload, { status: 204 })` thinking the `json` is just a status indicator. It is not — the body is sent.
- A `304` is built by reading the request headers and matching the ETag, but the handler still returns the body.
- A wrapper or middleware (e.g. an analytics shim) appends a body to every response.
- The handler returns `undefined` from a typed function, but TypeScript's `Response | undefined` union is silently coerced to an empty body in dev and a real body in prod.

The error is raised by the Swift-Rust server runtime (axum-based) before the response is flushed.

## Reproducing the error

```ts
import type { RequestHandler } from "swift-rust/server";

export const DELETE: RequestHandler = async () => {
  await db.items.delete(123);
  return new Response("deleted", { status: 204 });
};
```

## Error output

```
[ERROR] [SR0062] invalid API route status/body response
  --> app/api/items/[id]/route.ts:5:1
   |
   = status 204 does not allow a response body. the handler returned
     "deleted" (6 bytes).
   = hint: return `new Response(null, { status: 204 })` or pick a
           different status code.
```

## How to fix it

1. Use an explicit `null` body for `204`/`304`.

   ```ts
   import type { RequestHandler } from "swift-rust/server";

   export const DELETE: RequestHandler = async () => {
     await db.items.delete(123);
     return new Response(null, { status: 204 });
   };
   ```

2. If you need a body, pick a different status. `200 OK` is the usual choice for "deleted with a payload", and `200` allows a body.

   ```ts
   return Response.json({ ok: true }, { status: 200 });
   ```

3. For `304 Not Modified`, return an empty response with only the matching headers (`ETag`, `Cache-Control`).

   ```ts
   return new Response(null, {
     status: 304,
     headers: { etag: req.headers.get("if-none-match") ?? "" },
   });
   ```

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=server=trace bun swift-rust dev
curl -i -X DELETE http://localhost:3000/api/items/123
```

## Common pitfalls

- The `Response` constructor does not enforce the no-body rule. The framework enforces it at the server boundary, but a serverless deployment target may silently allow it.
- `Response.json(undefined)` is treated as `null` for HTTP status purposes; the runtime accepts it. `Response.json("anything")` is not.
- `304` requires an `ETag` (or `Last-Modified`) header. The runtime will warn separately if you forget it.
- This is the Swift-Rust analogue of Next.js's `invalid-api-status-body`. The fix is identical.

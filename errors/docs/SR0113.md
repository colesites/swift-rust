# SR0113 — Server import in page

A file imported from `swift-rust/server` outside of `proxy.ts`. The `swift-rust/server` module is reserved for the proxy/middleware layer; using it elsewhere is a structural mistake that the build refuses to compile.

## Why this error occurred

`swift-rust/server` exports `NextRequest`, `NextResponse`, and `NextFetchEvent` — the types and helpers for the proxy layer that runs in front of every request. The proxy is the only place that:

- Receives a mutable `NextRequest` it can rewrite.
- Sees `NextFetchEvent` for tracking request lifecycle.
- Can call `NextResponse.next()` to forward to the actual route.

These concepts do not exist inside a page, a layout, or a route handler. The request has already been routed by the time the handler runs, and a `NextRequest` is just a `Request` with some extra metadata. Importing the proxy types from a route handler is almost always a leftover from copy-pasting middleware code.

The build rejects the import so the mistake is caught at compile time rather than as a confusing runtime error.

## Reproducing the error

```ts filename="app/api/ping/route.ts"
import type { NextRequest, NextResponse } from "swift-rust/server"

export function GET(req: NextRequest) {
  return new Response("ok")
}
```

`NextRequest` is imported in a route handler, not in `proxy.ts`.

## Error output

```
error: [SR0113] server import in page: `swift-rust/server` may only be imported from proxy.ts
  --> app/api/ping/route.ts:1:25
   |
 1 | import type { NextRequest, NextResponse } from "swift-rust/server"
   |                         ^^^^^^^^^^ move this logic to proxy.ts
```

## How to fix it

1. **Move the logic to `proxy.ts`.** This is the right home for any logic that needs to inspect or rewrite the request before it reaches a route handler.

```ts filename="proxy.ts"
import { NextResponse, type NextRequest, type NextFetchEvent } from "swift-rust/server"

export function proxy(req: NextRequest, ev: NextFetchEvent) {
  if (!req.headers.get("authorization") && req.nextUrl.pathname.startsWith("/api")) {
    return new NextResponse("unauthorized", { status: 401 })
  }
  return NextResponse.next()
}
```

2. **If you only need a `Request` in a route handler.** Use the standard `Request` type from the Web Fetch API. The framework exposes the incoming request as the first argument to your handler.

```ts filename="app/api/ping/route.ts"
export function GET(req: Request) {
  const url = new URL(req.url)
  return new Response(`ok: ${url.pathname}`)
}
```

3. **If you need to redirect or rewrite from a page or layout.** Use the helpers from `swift-rust/navigation` (`redirect`, `permanentRedirect`) or `swift-rust/headers` (`headers`, `cookies`).

```tsx filename="app/dashboard/page.tsx"
import { redirect } from "swift-rust/navigation"
import { cookies } from "swift-rust/headers"

export default async function Page() {
  const session = (await cookies()).get("session")
  if (!session) redirect("/login")
  return <Dashboard />
}
```

## Diagnosis

```bash
bun swift-rust doctor --check server-imports
RUST_LOG=swift_rust=server=trace bun swift-rust build
```

`--check server-imports` lists every file that imports from `swift-rust/server` and whether the location is allowed.

## Common pitfalls

- Type-only imports (`import type { NextRequest } from "swift-rust/server"`) are still flagged. The structural rule is about the import site, not the runtime shape.
- The `swift-rust/server` module also exports `NextResponse.json()`, which is convenient in route handlers. The recommended import path is the bare `swift-rust/server` only from `proxy.ts`; in route handlers, use `NextResponse` from the same module but accept that the import site check still applies, or use plain `Response.json()`.
- A custom server entrypoint (`server.ts` in the project root) is allowed to import from `swift-rust/server`, since the entrypoint is what hosts the proxy.
- The check is static: a transitive import that surfaces the symbol in your code is still flagged at the file that does the import, not at the original definition.

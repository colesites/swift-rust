# SR0078 — Invalid custom route `source`

Swift-Rust uses `path-to-regexp` to parse the `source` field of rewrites, redirects, headers, and proxy `matcher` configs. The library is intentionally narrower than full `RegExp`: it accepts named placeholders, optional segments, and catch-alls, but it does not accept arbitrary regex syntax. When a pattern cannot be parsed, the framework raises this error pointing at the offending `source`.

## Why this error occurred

The most common offender is a negative lookahead, copied from a regex snippet. For example:

```json
{
  "source": "/feedback/(?!general)"
}
```

This is valid JavaScript regex, but it is not valid path-to-regexp. The library expects the lookahead to be wrapped in an unnamed parameter so the match can be captured and reused in the `destination`:

```json
{
  "source": "/feedback/((?!general).*)"
}
```

Other common causes:

- A `source` with a quantifier on a non-parameter, e.g. `/foo+`.
- A pattern with unbalanced parentheses.
- An invalid escape sequence (`\\d` is valid in regex but not in path-to-regexp).
- A custom regex that includes a `?` without a parameter context.

The error fires at server startup, before any request is served, so the dev server refuses to start.

## Reproducing the error

```json
{
  "rewrites": [
    {
      "source": "/feedback/(?!general)",
      "destination": "/feedback/general"
    }
  ]
}
```

```ts
// proxy.ts
import type { ProxyConfig } from "swift-rust/proxy";

export const config: ProxyConfig = {
  matcher: "/feedback/(?!general)",
};
```

## Error output

```
error: [SR0078] invalid custom route source
  --> swift-rust.config.json:3:5
   |
 3 |     {
 4 |       "source": "/feedback/(?!general)",
   |                          ^^^^^^^^^^^^^^^
   |
   = path-to-regexp does not support negative lookaheads. wrap the
     regex part in an unnamed parameter: "/feedback/((?!general).*)".
```

## How to fix it

1. Wrap regex-style atoms in an unnamed parameter.

   ```json
   {
     "rewrites": [
       {
         "source": "/feedback/((?!general).*)",
         "destination": "/feedback/general"
       }
     ]
   }
   ```

2. For proxy matchers, do the same.

   ```ts
   // proxy.ts
   import type { ProxyConfig } from "swift-rust/proxy";

   export const config: ProxyConfig = {
     matcher: "/feedback/((?!general).*)",
   };
   ```

3. If you need a full regex, use the `regex` field (only available in proxy `matcher` configs) and accept that the matcher cannot interpolate parameters.

   ```ts
   export const config: ProxyConfig = {
     matcher: { regex: "^/feedback/(?!general)(.*)$" },
   };
   ```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust route:list --rewrites --headers --redirects
RUST_LOG=swift_rust=routing=trace bun swift-rust dev
```

## Common pitfalls

- A `source` is parsed once at server start. If you change it, the dev server restarts automatically; in production you need a redeploy.
- The `matcher` field on proxy configs accepts both path-to-regexp (string) and full regex (`{ regex }`). Use the regex form when you genuinely need regex features.
- The framework does not warn about patterns that match "too much" — only about patterns that fail to parse. Audit your matchers with `bun swift-rust route:list` to make sure they are doing what you think.
- This is the Swift-Rust analogue of Next.js's `invalid-route-source`. The fix is identical.

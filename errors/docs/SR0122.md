# SR0122 — Parallel build enabled without a worker

The build pipeline detected that one of the parallel-build options is enabled, but the bundler worker that actually runs the work in parallel is not. The two options are silently inert until the worker is on, so Swift-Rust prints this informational message to make the situation visible.

## Why this error occurred

Swift-Rust can run server-side compilation and trace collection in parallel by forking worker processes. The orchestrator itself is a single Rust process, but it delegates the CPU-heavy work to a pool of workers. Two config keys control the parallelism:

- `build.parallelServerCompiles` — compiles multiple server bundles at once.
- `build.parallelServerBuildTraces` — runs the dependency trace for several entries in parallel.

Both options are only meaningful when the bundler worker is enabled (`build.bundlerWorker: true`). If the worker is off — usually because a custom bundler config is in place — the options do nothing. The CLI prints this message so the user is not surprised by the lack of speedup.

The message is informational; the build still completes.

## Reproducing the error

```json
{
  "build": {
    "bundlerWorker": false,
    "parallelServerCompiles": true,
    "parallelServerBuildTraces": true
  }
}
```

## Error output

```
[INFO] [SR0122] parallel build enabled without bundler worker
  --> swift-rust.config.json:2:5
    |
  2 |     "bundlerWorker": false,
  3 |     "parallelServerCompiles": true,
    |                  ^^^^^^^^^^^^^^^^
    | this option requires `build.bundlerWorker: true` to take effect.
    |
    help: set `build.bundlerWorker: true`, or remove the parallel options.
```

## How to fix it

1. Enable the bundler worker:

```json
{
  "build": {
    "bundlerWorker": true,
    "parallelServerCompiles": true,
    "parallelServerBuildTraces": true
  }
}
```

2. Alternatively, drop the parallel options if you must keep `bundlerWorker: false`:

```json
{
  "build": {
    "bundlerWorker": false
  }
}
```

3. Note that enabling the worker increases peak memory use because each worker is a separate process with its own bundler instance. Measure on the target machine before enabling both options at once.

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

`swift-rust info` shows which build flags are active and whether the worker is in use.

## Common pitfalls

- A custom `build.bundler` config disables the worker by default. If you need a custom bundler, you also have to manage the worker yourself.
- On memory-constrained CI runners, enabling both parallel options plus the worker can OOM. Start with one option, measure, then add the other.
- The parallel options affect build time, not runtime. They are useless in `bun swift-rust dev`.

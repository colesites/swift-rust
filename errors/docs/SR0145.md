# SR0145 — SWC compiler disabled

The SWC compiler (Rust-based) is the default for Swift-Rust. When the project contains a custom Babel configuration, SWC is disabled and the build falls back to Babel so that the existing transforms keep working. The build fails with this error if a `forceSwcTransforms: true` override was expected but a Babel config exists, or if the Babel fallback cannot satisfy a feature that the project relies on.

## Why this error occurred

Swift-Rust's default transforms — JSX, TypeScript stripping, server-component stripping, tree-shaking helpers — are all implemented in SWC. When you add a `.babelrc`, the framework opts out of SWC and hands the source to Babel so your plugins can run. This is the compatibility path for projects that need a Babel plugin that has no SWC equivalent.

The error is the strict version of the same condition: it fires when the build was told to use SWC (`forceSwcTransforms: true`) but a Babel config is present. The two are incompatible because Babel would pre-transform the source and the SWC pass would not have an opportunity to do its work.

## Reproducing the error

```json
{
  "compiler": {
    "forceSwcTransforms": true
  }
}
```

```js
// .babelrc
{
  "presets": ["@babel/preset-react"]
}
```

The config says "use SWC", the `.babelrc` says "use Babel", and the build errors out.

## Error output

```
[ERROR] [SR0145] SWC compiler disabled by Babel config
  --> .babelrc:1:1
    |
  1 | {
    | ^
    | a custom Babel configuration is present. SWC has been disabled.
    | if you want to force SWC, set `compiler.forceSwcTransforms: true` (but
    | then Babel plugins will not run).
    |
    help: port your Babel plugins to SWC, or remove `forceSwcTransforms: true` from the config.
```

## How to fix it

1. Port your Babel plugins to SWC equivalents. Most common presets (React, TypeScript, env) have a 1:1 SWC mapping.

2. If a direct port is not possible, accept the Babel fallback by removing the `forceSwcTransforms` override:

```json
{
  "compiler": {
    "forceSwcTransforms": false
  }
}
```

3. If you explicitly want to keep SWC and skip the Babel transforms, set the override and delete the `.babelrc`:

```bash
rm .babelrc
```

4. Re-run the build. The error should be gone.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=compiler=trace bun swift-rust dev
```

`compiler=trace` prints the resolved compiler (SWC vs. Babel), the source of the decision (config vs. `.babelrc` presence), and the list of active transforms.

## Common pitfalls

- A `babel.config.js` (project-wide) and a `.babelrc` (per-directory) are both detected. Pick one and delete the other.
- A dependency that ships a `babel.config.js` will be picked up if your project has no config. The error names the offending path; delete the dep's config or add your own.
- Babel and SWC do not produce identical output. Watch for differences in JSX runtime, decorator order, and import hoisting.
- `forceSwcTransforms: true` is a no-op in the absence of a `.babelrc` (SWC is already the default). The flag is only meaningful when both are present.

# SR0015 — Static-to-dynamic rendering error

A route in `app/` was generated as a fully static page at build time, but at runtime it tried to read request-only data (`cookies()`, `headers()`, `searchParams`, etc.). The Swift-Rust compiler refuses to switch a page from static to dynamic mid-flight because the two modes produce different caching and streaming behavior.

## Why this error occurred

The compiler decides at build time whether each page is fully prerenderable. If it is, the result is cached and served from a CDN. If the page later calls a request-time API, the cache is wrong: the cached HTML no longer reflects the live request. Rather than silently serving stale data, the runtime errors out.

The same problem appears in reverse when a dynamic page accidentally becomes static (a `revalidate` window of zero, a missing `await`, etc.). Both directions are covered under this code so the fix is always to make the mode intentional.

## Reproducing the error

```tsx
import { cookies } from "swift-rust/headers";

export default async function Page() {
  const jar = await cookies();
  return <p>theme = {jar.get("theme")?.value ?? "system"}</p>;
}
```

The page has no `dynamic` export and no `<Suspense>` around the read, so the compiler assumes it is prerenderable. The call to `cookies()` then breaks that assumption.

## Error output

```
[ERROR] [SR0015] static-to-dynamic rendering error: request API used in a static page
   --> app/profile/page.tsx:4:16
   |
 4 |   const jar = await cookies();
   |                ^^^^^^^^^^^^^^ this call requires a dynamic page
   |
  help: add `export const dynamic = 'force-dynamic'` to this file, or move the call below a <Suspense> boundary.
```

## How to fix it

1. Decide if the page really needs request data. If yes, force dynamic rendering explicitly.
2. If the page only needs the data in one branch, push the read into a child component and wrap that child in `<Suspense>` so the rest of the page stays static.
3. Rebuild and confirm the error is gone.

```tsx
import { cookies } from "swift-rust/headers";

export const dynamic = "force-dynamic";

export default async function Page() {
  const jar = await cookies();
  return <p>theme = {jar.get("theme")?.value ?? "system"}</p>;
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust build
```

The `build` trace lists every page classified as static and the request APIs it actually called during the prerender pass. Pages that crossed the line show up with a stack frame pointing at the offending call.

## Common pitfalls

- Reading `searchParams` counts as a request API. Pages that look static but accept query params must opt in.
- `generateStaticParams` does not make a page static by itself — it only narrows which params the build pre-renders.
- A `<Suspense>` boundary only helps if the dynamic read happens **inside** the boundary, not above it.
- This error is the build-time version of `SR0038` (dynamic server usage). Fixing one usually fixes the other.

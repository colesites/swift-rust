# SR0109 — Using <img> instead of <Image>

An `<img>` element was used in JSX. The framework would prefer `<Image>` from `swift-rust/image`, which handles responsive sizes, format negotiation, lazy loading, and the LCP image preload for you. This warning is informational — the image still renders — but the page is missing optimizations that are easy to add.

## Why this error occurred

A plain `<img>` sends the user a single, unoptimized image. The framework's `<Image>` does several things on top:

- Generates multiple variants at build time and serves the smallest one that fits the viewport.
- Negotiates modern formats (AVIF, WebP) based on the browser's `Accept` header.
- Reserves layout space with the `width` and `height` attributes to prevent layout shift.
- Preloads the LCP image so the browser can fetch it early.
- Lazy-loads below-the-fold images by default.

None of these are free, but the cumulative effect on LCP and bandwidth is large — often 30–50% smaller images and a measurably faster Largest Contentful Paint. The framework warns on `<img>` to make the omission visible during code review.

## Reproducing the error

```tsx filename="app/page.tsx"
export default function Page() {
  return (
    <img
      src="https://example.com/hero.jpg"
      alt="Landscape"
      width={800}
      height={500}
    />
  )
}
```

## How to fix it

1. **Recommended.** Replace with `<Image>` from `swift-rust/image` and configure the image host if it is remote.

```tsx filename="app/page.tsx"
import Image from "swift-rust/image"

export default function Page() {
  return (
    <Image
      src="https://example.com/hero.jpg"
      alt="Landscape"
      width={800}
      height={500}
    />
  )
}
```

2. **Want the API without the optimization.** Pass `unoptimized` to keep your current behavior while still getting the rest of the API.

```tsx filename="app/page.tsx"
import Image from "swift-rust/image"

const RawImage = (props: React.ComponentProps<typeof Image>) => (
  <Image {...props} unoptimized />
)
```

3. **Format fallback only.** Use `<picture>` with multiple `<source>` elements. This is the right call when you control the source set and don't need the framework's optimizer.

```tsx filename="app/page.tsx"
export default function Page() {
  return (
    <picture>
      <source srcSet="https://example.com/hero.avif" type="image/avif" />
      <source srcSet="https://example.com/hero.webp" type="image/webp" />
      <img
        src="https://example.com/hero.jpg"
        alt="Landscape"
        width={800}
        height={500}
      />
    </picture>
  )
}
```

4. **Custom loader.** Provide a custom loader file and reference it from `swift-rust.config.json`.

```js filename="lib/loader.js"
export default function loader({ src, width, quality }) {
  return `https://cdn.example.com/${src}?w=${width}&q=${quality ?? 75}`
}
```

```json filename="swift-rust.config.json"
{
  "image": {
    "loader": "custom",
    "loaderFile": "./lib/loader.js"
  }
}
```

## Diagnosis

```bash
bun swift-rust doctor --check images
bun swift-rust info --images
```

`--check images` lists every `<img>` in `app/` and the closest `<Image>` equivalent.

## Common pitfalls

- The warning is suppressed in production builds by default; it is a development-time signal. If you want it in CI, run `bun swift-rust doctor --check images --strict`.
- A `<picture>` element with a nested `<img>` is a different (and valid) pattern; the warning still fires on the inner `<img>`, which is correct.
- Self-hosting requires `sharp` to be installed. The first-party Swift-Rust platform installs it automatically.
- Each unique quality produces a cached variant. The image quality error ([SR0100](SR0100.md)) is what constrains the set.

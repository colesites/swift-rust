# SR0111 — Router.onAppUpdated has been removed

The code references `Router.onAppUpdated`, a hook that was removed years ago. The hook existed in early versions of Next.js to let apps react to a new deployment being detected during client navigation, and Swift-Rust never supported it. The deprecation warning surfaces during code review so the broken reference can be removed.

## Why this error occurred

In Next.js 9–10, `Router.onAppUpdated` was a hook that fired when the client noticed a new app deployment. It was often used to navigate to a relative URL, which would force the new code to load. The mechanism was limited: it could not await a network request, and it could not block navigation, so its real use cases were narrow.

The hook was removed in Next.js 11 because the framework's own deployment detection is now good enough that you do not need to intervene. Swift-Rust inherits the same model. If you genuinely need to persist state on a new deployment, the closest equivalent is `window.onbeforeunload`, which fires before the page unloads (including across deploys).

## Reproducing the error

```ts filename="app/bootstrap.ts"
import Router from "swift-rust/router"

Router.onAppUpdated = function (nextRoute) {
  window.location.href = nextRoute
}
```

## Error output

```
warn: [SR0111] Router.onAppUpdated has been removed
  --> app/bootstrap.ts:3:1
   |
 3 | Router.onAppUpdated = function (nextRoute) {
   | ^^^^^ this hook was removed; the framework detects new deployments automatically
```

## How to fix it

1. **Persist state before unload.** If your real goal was saving application state to local storage, move that into `window.onbeforeunload`.

```ts filename="app/bootstrap.ts"
window.onbeforeunload = function () {
  const state = { /* gather your store here */ }
  localStorage.setItem("app-state", JSON.stringify(state))
}
```

2. **Re-fetch on focus/visibility.** If your real goal was "re-fetch when the user comes back to the tab", the modern equivalent is to subscribe to `visibilitychange` and call your data library's revalidation hook (e.g. SWR's `useSWRConfig`).

```ts filename="app/bootstrap.ts"
import { mutate } from "swr"

document.addEventListener("visibilitychange", () => {
  if (document.visibilityState === "visible") {
    void mutate(() => true, undefined, { revalidate: true })
  }
})
```

3. **Delete the file.** If neither of the above is needed, just delete the bootstrap script. The framework detects new deployments for you.

## Diagnosis

```bash
bun swift-rust doctor --check deprecations
RUST_LOG=swift_rust=router=trace bun swift-rust dev
```

`--check deprecations` lists every removed or renamed API in your codebase.

## Common pitfalls

- The `Router` import from `swift-rust/router` is itself a legacy path. The modern equivalent is `useRouter` from `swift-rust/navigation`. A code that imports the default and assigns to a property on it is doubly outdated.
- The deprecation is a warning, not an error. The build still succeeds. The file is dead at runtime, so the warning will become a real "this code does nothing" bug in your application.
- The framework's own deployment detection uses a build-time ID embedded in the HTML. The client compares IDs on every navigation and triggers a hard reload if they differ. You do not need to do anything to opt in.

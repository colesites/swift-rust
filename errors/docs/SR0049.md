# SR0049 — Fast refresh had to perform a full reload

Fast Refresh is the dev-time mechanism that re-runs a component without losing its state when you save a file. It works by tracking the React component exports of each module and replacing them in place. Swift-Rust falls back to a full page reload when it cannot identify a single React component to patch in isolation, so the page state is lost but the change is still applied.

## Why this error occurred

Fast Refresh has a strict contract: a module is "refresh-safe" only when it exports a single React component and nothing else (besides the standard Segment Configuration Options like `revalidate` or `metadata`). The moment a module also exports a helper function, a constant, or a type, Fast Refresh has to reload the page because it cannot know which downstream modules also depend on those exports.

Other common reasons a reload is forced:

- The component is an anonymous default export: `export default function () { … }`. Fast Refresh needs a name to display in the error overlay and to map HMR boundaries.
- The component is named in `camelCase` or `snake_case` rather than `PascalCase`. This is a soft heuristic — JSX `<textfield />` is indistinguishable from a built-in HTML element, so the linter and Fast Refresh refuse to treat it as a component.
- A class component is exported. React Server Components and modern React patterns prefer functions; class components are still supported but cannot be hot-patched.
- A file that defines a component also re-exports a hook or a constant from a barrel file.

## Reproducing the error

```tsx
export const helper = (n: number) => n * 2;

export default function () {
  return <div>{helper(2)}</div>;
}
```

## Error output

```
[INFO] [SR0049] fast refresh had to perform a full reload
  --> app/page.tsx
   |
   = the module exports values other than a React component, so the
     hot-update boundary cannot be safely established.
   = state will be lost on the next save. save again to recover.
```

## How to fix it

1. Move helper exports to a sibling file.

   ```tsx
   // lib/math.ts
   export const helper = (n: number) => n * 2;
   ```

2. Convert the component to a named function in `PascalCase`.

   ```tsx
   import { helper } from "../lib/math";

   export default function Page() {
     return <div>{helper(2)}</div>;
   }
   ```

3. If you genuinely need the helper co-located, give it a non-component name and accept that the file will not be hot-reloadable.

## Diagnosis

```bash
bun swift-rust doctor
RUST_LOG=swift_rust=hmr=debug bun swift-rust dev
```

## Common pitfalls

- Anonymous default exports from a function expression are detected at edit time. The fix is to name the function, not the variable.
- Barrel files (`index.ts` that re-exports many modules) always force a full reload on edit. Keep barrels thin or avoid them.
- `useState` and `useRef` state are preserved across refreshes when the contract above is respected. Anything stored in module scope (e.g. `let counter = 0`) is reset on every refresh regardless of reload vs. hot-patch.
- The `info` severity means the build did not fail. Treat the message as guidance, not a stop-the-world error.

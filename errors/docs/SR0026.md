# SR0026 — Uncaught client-side exception

A client component in production threw an error that no error boundary caught. The runtime caught it, marked the boundary missing, and emitted this code so the failure is visible in the error overlay (dev) or the production error tracker.

## Why this error occurred

Client React trees that are not wrapped in an error boundary are fragile: any throw in render or in an effect unmounts the whole subtree. The runtime catches the error to keep the rest of the page alive but logs it as `SR0026` so the missing boundary is obvious.

In production, the user sees a partial page and no obvious indicator unless your error tracker is wired up. In dev, the error overlay prints the code so the boundary is easy to find.

## Reproducing the error

```tsx
"use client";

export default function Cart({ items }: { items: { price: number }[] }) {
  const total = items.reduce((acc, i) => acc + i.price, 0);
  return <p>total = {total.toFixed(2)}</p>;
}
```

If `items` is `undefined` (for example, when a parent forgets to pass it), the `reduce` call throws and there is no error boundary in `app/page.tsx`.

## Error output

```
[ERROR] [SR0026] uncaught client-side exception
   component: Cart (app/components/cart.tsx:3:1)
   error: TypeError: Cannot read properties of undefined (reading 'reduce')
   |
   help: wrap the component in an <ErrorBoundary> with a fallback UI, or fix the data source.
```

## How to fix it

1. Add a React error boundary above the failing component with a sensible fallback.
2. For event handlers, wrap the body in `try/catch` and surface a friendly message.
3. Reproduce the same call in dev (`bun swift-rust dev`) to read the source-mapped stack frame and pinpoint the throw.

```tsx
"use client";
import { Component, type ReactNode } from "react";

export default class CartBoundary extends Component<
  { children: ReactNode },
  { error?: Error }
> {
  state = { error: undefined as Error | undefined };
  static getDerivedStateFromError(error: Error) {
    return { error };
  }
  render() {
    if (this.state.error) return <p>cart unavailable</p>;
    return this.props.children;
  }
}
```

```tsx
import CartBoundary from "./components/cart-boundary";
import Cart from "./components/cart";

export default function Page() {
  return (
    <CartBoundary>
      <Cart items={[]} />
    </CartBoundary>
  );
}
```

## Diagnosis

```bash
bun swift-rust doctor
bun swift-rust info
RUST_LOG=swift_rust=trace bun swift-rust dev
```

The trace log prints the error message, component name, and source frame. In production, the same data is forwarded to the configured error tracker.

## Common pitfalls

- The error overlay only appears in dev. In production, the only signal is the production error tracker; make sure it is wired up.
- An error boundary must be **above** the failing component. Placing it as a sibling does not help.
- Async errors thrown from event handlers do not bubble to React; they need an explicit `try/catch`.
- This error and `SR0017` are different: `SR0017` is about prerender-time data access, `SR0026` is about runtime client throws.

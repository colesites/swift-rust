## Why this error occurred

The native `<video>` element emits a `MediaError` event whenever the load fails. The browser exposes four `MediaError.code` values:

| Code | Meaning | `VideoError.kind` |
|---|---|---|
| 1 | `MEDIA_ERR_ABORTED` — the user aborted the load | `"media"` |
| 2 | `MEDIA_ERR_NETWORK` — network error while loading | `"network"` |
| 3 | `MEDIA_ERR_DECODE` — the file is corrupt or uses an unsupported codec | `"decode"` |
| 4 | `MEDIA_ERR_SRC_NOT_SUPPORTED` — the source format is not supported | `"src-not-supported"` |

The viewer maps these into the `VideoError.kind` field. The `mediaErrorCode` field preserves the original browser code so you can dispatch on it.

## Reproducing the error

```tsx
<Video src="/missing.mp4" />
```

The error UI replaces the player. The browser console logs:

```
VideoError [SR0154]: Source format not supported
    at <VideoSurface> (packages/video/src/index.ts:418:7)
```

## Error output

```
VideoError: src-not-supported
  code: "SR0154"
  kind: "src-not-supported"
  mediaErrorCode: 4
  url: "/missing.mp4"
```

The viewer also reports a `"decode"` error (`mediaErrorCode: 3`) if the file is a video but the codec is unsupported (e.g. ProRes on a browser without a ProRes decoder).

## How to fix it

1. **Verify the URL.** Open it in a new tab. If the browser does not play it directly, the URL is wrong or the file is malformed.

2. **Provide multiple sources.** Browsers pick the first source they support:

   ```tsx
   <Video
     src={[
       { src: "/clip.webm", type: "video/webm" },
       { src: "/clip.mp4", type: "video/mp4" },
     ]}
   />
   ```

   The viewer resolves the `type` for you if you omit it (using the file extension), but explicit is safer for cross-origin sources where the server's `Content-Type` may differ from the file's actual contents.

3. **Re-encode unsupported codecs.** HEVC (H.265), ProRes, and AV1 are not universally supported. H.264 + AAC in an MP4 container is the only format that works in every browser.

4. **Add CORS headers for cross-origin videos.** The server must respond with `Access-Control-Allow-Origin: <your-origin>` and the video must be served with `Accept-Ranges: bytes` if you want seeking to work.

5. **Treat user-cancelled loads as non-fatal.** A `kind: "media"` with `mediaErrorCode: 1` means the user aborted. Do not surface it as an error to them.

## Diagnosis

```tsx
<Video
  src="/clip.mp4"
  onError={(err) => {
    if (err.kind === "network") analytics("video.network");
    if (err.kind === "decode") analytics("video.decode");
    if (err.kind === "src-not-supported") analytics("video.unsupported");
    if (err.mediaErrorCode === 1) return;
  }}
/>
```

The `mediaErrorCode` is the raw browser code (1–4); the `kind` is the viewer-mapped category. Dispatch on whichever is more convenient.

## Common pitfalls

- **Mixing `autoPlay` with `muted: false`.** Most browsers refuse to autoplay an unmuted video. Either set `muted` (and add an "Unmute" button) or trigger playback from a user gesture.
- **Using a CDN that serves wrong MIME types.** Cloudflare R2 and some S3 configurations serve everything as `application/octet-stream`. The `<video>` element will refuse to play it on some browsers. Set `Content-Type` explicitly when uploading.
- **Not handling the player-removing the source after a `src` change.** When you change the `src` prop, the viewer tears down the previous `<video>` and renders a new one. If you want a seamless transition, preload the next source yourself.
- **Treating `MEDIA_ERR_ABORTED` as a fatal error.** It is the user pressing Stop, or React unmounting the player. Do not show an error UI for it.

# SR0080 — Invalid layout or page export

Swift-Rust is strict about what a file in `app/` can export. The convention is that every file in the route tree (`layout.tsx`, `page.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`, `template.tsx`, `default.tsx`, `route.ts`) can export only:

- A **default export** — the React component that the framework will render for that segment.
- A fixed list of **named segment configuration exports** — `revalidate`, `dynamic`, `dynamicParams`, `fetchCache`, `runtime`, `preferredRegion`, `maxDuration`, `unstable_instant`, and a handful of experimental flags.

Anything else is rejected with this error. The whitelist exists so the framework can detect misspelt configuration options and prevent conflicts with future features.

## Why this error occurred

Common causes:

- A developer adds a helper function to `page.tsx` and exports it for testing. The framework only looks for the default export and a small set of known names; the helper is treated as an unknown export.
- A custom config object is exported with a name that does not match the segment config spec, e.g. `export const myConfig = { … }`.
- A page re-exports a component from another module (`export { Page } from './page-impl'`) without defining its own default export.
- A barrel file (`index.ts`) re-exports many things and lives inside the `app/` tree, where exports are restricted.

The error fires at the next dev rebuild, or at the start of `bun swift-rust build`. It is raised against the offending line in the file that should have declared only the default component and the segment config.

## Reproducing the error

```tsx
// app/dashboard/page.tsx
export const helper = (n: number) => n * 2;

export default function Page() {
  return <div>{helper(2)}</div>;
}
```

## Error output

```
error: [SR0080] invalid layout or page export
  --> app/dashboard/page.tsx:2:1
   |
 2 | export const helper = (n: number) => n * 2;
   | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
   |
   = `helper` is not a recognized export for an app/page file. only
     the default React component and the segment configuration
     options are allowed. move `helper` to a sibling file.
```

## How to fix it

1. Move the helper to a sibling file.

   ```ts
   // app/dashboard/helpers.ts
   export const helper = (n: number) => n * 2;
   ```

   ```tsx
   // app/dashboard/page.tsx
   import { helper } from "./helpers";

   export default function Page() {
     return <div>{helper(2)}</div>;
   }
   ```

2. Use the standard segment config options for any per-route metadata.

   ```tsx
   export const revalidate = 60;
   export const dynamic = "force-dynamic";

   export default function Page() {
     return <div>Hello</div>;
   }
   ```

3. If a custom export is genuinely needed (e.g. for an internal library), co-locate it in a file outside the `app/` tree, or in a `components/` subfolder that is not part of the route table.

## Diagnosis

```bash
bun swift-rust doctor
rg -n "^export " app/
RUST_LOG=swift_rust=app=trace bun swift-rust build
```

## Common pitfalls

- The default export can be named anything — `Page`, `default`, even a string if you really want. The framework reads the file and looks for `export default`.
- Re-exports are not allowed in route files: `export { config } from './shared'` and `export * from './shared'` are both flagged.
- The `metadata` export is the only "exotic" export allowed in `app/`. Anything else has to live outside the file.
- A barrel file (`index.ts`) that re-exports many modules is the most common cause in a freshly-migrated codebase. Replace barrels with direct imports in the route files.
- This is the Swift-Rust analogue of Next.js's `invalid-segment-export`. The fix is identical.
